error handling

TRY

When things go wrong — and they will — Python gives you a safety net. Catch errors gracefully, recover from them, and keep your program running.

// section 01

The four clauses

A full error-handling block can have up to four parts. Only try and except are required — else and finally are optional but powerful.

try — the risky code
What you put here: any code that might raise an error. Python runs this first. If no error occurs, the except block is skipped entirely.
except — the safety net
What you put here: code that handles the error. You can catch specific exception types, catch multiple, and access the error object with as e.
else — only if no error
What you put here: code that should run only when the try block succeeded without any exception. Keeps "success" logic clean and separate.
finally — always runs
What you put here: cleanup code that must run no matter what — error or not. Perfect for closing files, releasing connections, or freeing resources.

// section 02

Basic syntax

Start simple: wrap the risky code, catch the error, and handle it.

basic.py
# ── Simplest form ──────────────────────────────────
try:
    number = int(input("Enter a number: "))
    print(f"You entered: {number}")

except ValueError:
    print("⚠️  That's not a valid number!")

# ── With 'as e' to inspect the error ───────────────
try:
    result = 10 / 0

except ZeroDivisionError as e:
    print(f"Error caught: {e}")   # division by zero

Execution flow

How Python decides which block runs depends on whether an exception is raised inside try.

try: block runs
exception raised?
YES ✗
except: block runs
NO ✓
else: block runs
finally: ALWAYS runs
program continues

// section 04

Catching multiple exceptions

You can stack multiple except clauses to handle different errors differently — or catch several at once using a tuple.

multiple_except.py
def safe_divide(a, b):
    try:
        result = a / b

    except ZeroDivisionError:
        print("Cannot divide by zero")

    except TypeError:
        print("Both arguments must be numbers")

    else:
        print(f"Result: {result}")   # only runs if no error

    finally:
        print("--- done ---")         # always runs

# ── Catch multiple types at once ───────────────────
try:
    data = int(input("Enter: "))

except (ValueError, TypeError) as e:
    print(f"Input error: {e}")

Full real-world example

A safe file reader that handles every likely failure point gracefully.

safe_read.py
def read_score(filename):
    """Read an integer score from a file, handling all errors."""
    file = None

    try:
        file = open(filename, "r")
        content = file.read().strip()
        score = int(content)

    except FileNotFoundError:
        print(f"❌ File '{filename}' not found.")
        return None

    except ValueError:
        print(f"❌ File content is not a valid number.")
        return None

    else:
        print(f"✅ Score loaded: {score}")
        return score

    finally:
        if file:
            file.close()          # always close the file!
            print("📁 File closed.")

Best practices

✅
Catch specific exceptions
Always name the exception type: except ValueError. This way you only catch what you expect — other errors still propagate and surface properly.
⚠️
Avoid bare except:
except: with no type catches everything — including keyboard interrupts and system exits. This can hide serious bugs and make debugging a nightmare.
💡
Use else for "success only" logic
Put code that should only run on success in else, not inside try. This keeps the try block minimal and avoids accidentally catching errors you didn't intend to.
🔒
Use finally for cleanup
Files, database connections, and network sockets should always be closed. Put that in finally — or better yet, use a with statement which handles it automatically.
🔕
Don't silently swallow errors
An empty except block that does nothing (pass) hides problems silently. At minimum, log the error or print a message so you know something went wrong.
❌
Don't use try/except for flow control
Avoid using exceptions as an alternative to if checks. Exceptions are for unexpected errors, not normal logic. They're slower and make code harder to understand.

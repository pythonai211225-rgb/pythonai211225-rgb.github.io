Lesson 6 · Normalization

Third Normal Form 3NF

A table is in 3NF when it is in 2NF and every non-key column depends only on the primary key — never on another non-key column.

Transitive Dependency

A transitive dependency exists when:

PK → column A → column B

Column B doesn't directly depend on the PK — it depends on column A, which itself depends on the PK. That creates an indirect (transitive) path.

Why it's a problem
If dept_id → dept_name, then dept_name is stored in every employee row. Rename the department in one row and other rows still show the old name — update anomaly.

❌ Bad Example — Violates 3NF

This employees table is in 2NF but violates 3NF because dept_name depends on dept_id, not on id:

id (PK) name salary dept_id dept_name ❌ dept_budget ❌
1Alice9000010 Engineering500000
2Bob8500010 Engineering500000
3Carol7500020 Marketing200000
4Dave7000020 Marketing200000
Redundancy: "Engineering" and 500000 are stored in two rows. If Engineering changes its name, you must update every row.
Insertion anomaly: You can't add a new department until you hire an employee for it — because dept_name only lives inside employee rows.
Deletion anomaly: If you delete all employees in a department, you lose the department's name and budget entirely.

✅ Good Example — Conforms to 3NF

Extract the transitively-dependent columns into their own table:

departments

id (PK)namebudget
10Engineering500000
20Marketing200000

employees

id (PK)namesalarydept_id (FK)
1Alice9000010
2Bob8500010
3Carol7500020
4Dave7000020
Renaming "Engineering" now requires updating exactly one row in departments. You can also create a department with zero employees. Deleting all employees from a department doesn't destroy the department's data.

SQL — 3NF Schema

CREATE TABLE departments (
  id     INTEGER  PRIMARY KEY AUTOINCREMENT,
  name   TEXT     NOT NULL UNIQUE,
  budget REAL     NOT NULL DEFAULT 0
);

CREATE TABLE employees (
  id      INTEGER  PRIMARY KEY AUTOINCREMENT,
  dept_id INTEGER  NOT NULL,
  name    TEXT     NOT NULL,
  salary  REAL     NOT NULL DEFAULT 0,
  FOREIGN KEY (dept_id) REFERENCES departments(id) ON DELETE RESTRICT
);

Sample rows — department data stored once, employees reference it by id:

INSERT INTO departments (name, budget) VALUES
  ('Engineering', 120000),
  ('Marketing',   80000),
  ('HR',          60000);

INSERT INTO employees (dept_id, name, salary) VALUES
  (1, 'Alice',   95000),   -- Engineering
  (1, 'Bob',     88000),   -- Engineering
  (2, 'Carol',   72000),   -- Marketing
  (3, 'Dave',    65000);   -- HR

Querying with department name via JOIN:

SELECT
  e.name     AS employee,
  d.name     AS department,
  d.budget,
  e.salary
FROM employees e
JOIN departments d ON d.id = e.dept_id
ORDER BY d.name, e.name;

Normalization Cheat Sheet

1NF
Atomic values. One value per cell, no repeating column groups, every row unique (has a PK).
2NF
No partial dependencies. Must be 1NF. Every non-key column depends on the whole composite PK — not just part of it.
3NF
No transitive dependencies. Must be 2NF. Non-key columns must depend only on the PK — not on other non-key columns.
Practical rule of thumb: "Every non-key column must depend on the key, the whole key, and nothing but the key." (adapted from the famous SQL saying)

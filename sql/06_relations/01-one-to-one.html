Lesson 6 · Relationships

One-to-One 1:1

Each row in Table A corresponds to at most one row in Table B — and each row in B corresponds to at most one row in A.

When to use 1:1

A 1:1 relationship splits one large entity into two tables. Common reasons:

  • Keep optional / rarely-accessed columns separate to speed up the main table
  • Isolate sensitive data (e.g. passwords, biometric info) in a restricted table
  • Extend a table you cannot modify (e.g. a vendor-supplied schema)
Real-world examples
users ↔ user_profiles
employees ↔ contracts
products ↔ product_details
accounts ↔ account_settings

Key rule: In user_profiles, the column user_id is both the PRIMARY KEY and a FOREIGN KEY referencing users(id). The PK constraint automatically ensures UNIQUE — so one user can have exactly one profile.

SQL Syntax

Shared primary key (most common for 1:1)

CREATE TABLE users (
  id    INTEGER  PRIMARY KEY AUTOINCREMENT,
  name  TEXT     NOT NULL,
  email TEXT     UNIQUE NOT NULL
);

CREATE TABLE user_profiles (
  user_id    INTEGER  PRIMARY KEY,  -- PK ensures uniqueness
  bio        TEXT,
  avatar_url TEXT,
  FOREIGN KEY (user_id) REFERENCES users(id)
    ON DELETE CASCADE
);

Step-by-Step Example

Step 1 — Insert into the parent table first

INSERT INTO users (name, email)
VALUES
  ('Alice', 'alice@example.com'),
  ('Bob',   'bob@example.com'),
  ('Carol', 'carol@example.com');

Step 2 — Insert into the child table using the parent's id

INSERT INTO user_profiles (user_id, bio, avatar_url)
VALUES
  (1, 'Love hiking and SQL', '/img/alice.png'),
  (2, 'Backend developer',   '/img/bob.png');

Step 3 — Query with INNER JOIN (only users that HAVE a profile)

SELECT
  u.id, u.name, u.email,
  p.bio, p.avatar_url
FROM users u
INNER JOIN user_profiles p ON p.user_id = u.id;

Step 4 — LEFT JOIN to include users without a profile

SELECT
  u.name,
  COALESCE(p.bio, 'No profile yet') AS bio
FROM users u
LEFT JOIN user_profiles p ON p.user_id = u.id;

Exercise

Build a Citizens ↔ ID Cards schema

  1. Create a citizens table with columns: citizen_id (PK AUTOINCREMENT), full_name TEXT NOT NULL, city TEXT.
  2. Create an id_cards table with citizen_id as both PK and FK referencing citizens(citizen_id) with ON DELETE CASCADE. Also add card_number TEXT UNIQUE NOT NULL and expires TEXT.
  3. Insert 3 citizens: Sophia Martinez (Barcelona), James Chen (Toronto), Amira Hassan (Cairo).
  4. Insert 2 cards: Sophia (ID-2024-5521, expires: 2032-12-15) and James (ID-2023-7744, expires: 2031-08-22). Leave Amira without a card.
  5. Write an INNER JOIN query to list citizens with their card number.
  6. Write a LEFT JOIN query to show ALL citizens — displaying 'No card' for those without one.
  7. Try inserting a second id card for citizen id 1. What error do you get and why?
  8. Delete a citizen (who has an id card). Then query the id_cards table. What happened to their card? Why?

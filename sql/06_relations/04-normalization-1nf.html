Lesson 6 · Normalization

First Normal Form 1NF

The foundation of normalization. A table is in 1NF when every column holds atomic (indivisible) values, there are no repeating groups, and each row can be uniquely identified.

The 1NF Rules

①
Atomic values
Each cell contains one single value — not a list, not a comma-separated string, not JSON.
②
No repeating groups
Don't use columns like phone1, phone2, phone3. Store multiple values as separate rows in a related table.
③
Unique rows
Every row must be uniquely identifiable — typically via a PRIMARY KEY.
④
Consistent column types
All values in a column must be of the same kind (e.g., all dates, all numbers, all names).

❌ Bad Example — Violates 1NF

This students table stores multiple phone numbers in one cell and has repeating column groups:

idnamephone_numberssubject1subject2subject3
1Alice 555-1111, 555-2222 MathScienceNULL
2Bob 555-3333, 555-4444, 555-5555 HistoryArtPE
Problem 1 — Multiple values in one cell: How do you query "students with phone 555-2222"? You'd need LIKE '%555-2222%' — fragile and slow.
Problem 2 — Repeating groups: What if a student has 4 subjects? You'd need to ALTER TABLE and add a column. This is a design smell.
Problem 3 — NULL waste: Alice's subject3 is NULL — wasted space for columns that don't apply.

✅ Good Example — Conforms to 1NF

Split multi-valued and repeating columns into separate related tables:

students

2Bob

Now every query is clean: SELECT phone FROM student_phones WHERE student_id = 1. Adding a 4th subject for Bob requires just one new row — no schema changes needed.

SQL — Creating 1NF Tables

CREATE TABLE students (
  id   INTEGER  PRIMARY KEY AUTOINCREMENT,
  name TEXT     NOT NULL
);

CREATE TABLE student_phones (
  id         INTEGER  PRIMARY KEY AUTOINCREMENT,
  student_id INTEGER  NOT NULL,
  phone      TEXT     NOT NULL,
  FOREIGN KEY (student_id) REFERENCES students(id) ON DELETE CASCADE
);

CREATE TABLE student_subjects (
  student_id INTEGER  NOT NULL,
  subject    TEXT     NOT NULL,
  PRIMARY KEY (student_id, subject),
  FOREIGN KEY (student_id) REFERENCES students(id) ON DELETE CASCADE
);

Sample rows — one value per cell, each relationship in its own table:

INSERT INTO students (name) VALUES
  ('Alice'),
  ('Bob');

INSERT INTO student_phones (student_id, phone) VALUES
  (1, '555-1111'),
  (1, '555-2222'),   -- Alice has two phones
  (2, '555-3333');

INSERT INTO student_subjects (student_id, subject) VALUES
  (1, 'Math'),
  (1, 'Physics'),
  (2, 'Math'),
  (2, 'History');

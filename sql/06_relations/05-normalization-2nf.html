Lesson 6 · Normalization

Second Normal Form 2NF

A table is in 2NF when it is already in 1NF and every non-key column depends on the entire primary key — not just part of it.

Partial Dependency

A partial dependency occurs when a non-key column depends on only part of a composite primary key.

2NF only matters when the primary key is composite (made of two or more columns). A table with a single-column PK is automatically in 2NF (if it's already in 1NF).

The fix
Move partially-dependent columns into a new table where the part of the key they depend on becomes the PK of that new table.
composite PK
student_id + course_id
→
fully dependent
grade depends on BOTH student + course
≠

student_name depends only on student_id

Now renaming "SQL Fundamentals" requires updating exactly one row in courses. The grade is the only column in enrollments that genuinely depends on the combination of student + course — proper 2NF.

SQL — 2NF Schema

CREATE TABLE students (
  id   INTEGER  PRIMARY KEY AUTOINCREMENT,
  name TEXT     NOT NULL
);

CREATE TABLE courses (
  id    INTEGER  PRIMARY KEY AUTOINCREMENT,
  title TEXT     NOT NULL UNIQUE
);

CREATE TABLE enrollments (
  student_id INTEGER  NOT NULL,
  course_id  INTEGER  NOT NULL,
  grade      TEXT,
  PRIMARY KEY (student_id, course_id),
  FOREIGN KEY (student_id) REFERENCES students(id),
  FOREIGN KEY (course_id)  REFERENCES courses(id)
);

Sample rows — student and course data stored once, linked via enrollments:

INSERT INTO students (name) VALUES
  ('Alice'),
  ('Bob');

INSERT INTO courses (title) VALUES
  ('Math 101'),
  ('History 201'),
  ('Physics 301');

INSERT INTO enrollments (student_id, course_id, grade) VALUES
  (1, 1, 'A'),   -- Alice · Math 101
  (1, 3, 'B'),   -- Alice · Physics 301
  (2, 1, 'C'),   -- Bob   · Math 101
  (2, 2, 'A');   -- Bob   · History 201

Querying with names is still easy — just JOIN:

SELECT s.name, c.title, e.grade
FROM  enrollments e
JOIN  students s ON s.id = e.student_id
JOIN  courses  c ON c.id = e.course_id;

Triggers — Automate Your Database

A trigger is SQL that fires automatically when data changes. Insert a grade and watch the class summary update itself — no extra code, no manual syncing.

What Is a Trigger?

A trigger watches a table. The moment a specified event happens — INSERT, UPDATE, or DELETE — the trigger runs SQL you defined. You never call it manually.

INSERT / UPDATE / DELETE
→
TRIGGER fires
→
SQL runs automatically

Trigger timing:

  • BEFORE — runs before the change. Use for validation — call RAISE(ABORT, …) to reject bad data.
  • AFTER — runs after the change lands. Use for syncing summary tables or logging.

Access the row inside the trigger:

  • NEW.column — the incoming row's values (INSERT, UPDATE)
  • OLD.column — the row being replaced (UPDATE, DELETE)

The Trigger — AFTER INSERT on grades

Every time a grade is inserted, this trigger recalculates the COUNT, AVG, MIN, and MAX for that class and writes them into class_stats — using NEW.class_id to target only the right class.

CREATE TRIGGER
CREATE TRIGGER update_class_stats
AFTER INSERT ON grades
FOR EACH ROW
BEGIN
  UPDATE class_stats
  SET
    student_count = (SELECT COUNT(*) FROM grades
                     WHERE class_id = NEW.class_id),
    avg_grade     = (SELECT ROUND(AVG(grade),1) FROM grades
                     WHERE class_id = NEW.class_id),
    min_grade     = (SELECT MIN(grade) FROM grades
                     WHERE class_id = NEW.class_id),
    max_grade     = (SELECT MAX(grade) FROM grades
                     WHERE class_id = NEW.class_id)
  WHERE class_id = NEW.class_id;
END;

💡 NEW.class_id is the class_id of the row just inserted. FOR EACH ROW means the trigger fires once per inserted row — if you insert 3 grades in one batch, the trigger runs 3 times.

BEFORE Trigger — Reject Bad Data

Use a BEFORE trigger to validate data before it lands. RAISE(ABORT, …) cancels the INSERT and rolls back.

Validation trigger — grade must be 0–100
CREATE TRIGGER validate_grade
BEFORE INSERT ON grades
FOR EACH ROW
BEGIN
  SELECT RAISE(ABORT, 'Grade must be between 0 and 100')
  WHERE  NEW.grade < 0 OR NEW.grade > 100;
END;

If you try INSERT INTO grades VALUES (99, 1, 'Test', 150) — the trigger fires, RAISE aborts the whole statement, and no row is written. The table stays clean.

Dropping a Trigger

SQL
DROP TRIGGER IF EXISTS update_class_stats;
DROP TRIGGER IF EXISTS validate_grade;

Dropping a trigger does NOT delete any table data. Only the automation is removed.

Build a trigger for an inventory system

products & inventory_log
products: id · name · stock_count  |  inventory_log: id · product_id · change · changed_at
exercise setup
hover to reveal SQL ↓
CREATE TABLE products (
  id          INTEGER PRIMARY KEY,
  name        TEXT,
  stock_count INTEGER DEFAULT 0
);
CREATE TABLE inventory_log (
  id         INTEGER PRIMARY KEY,
  product_id INTEGER,
  change     INTEGER, -- positive = restock, negative = sale
  changed_at TEXT
);
INSERT INTO products (name, stock_count) VALUES
  ('Laptop',  20),
  ('Phone',   50),
  ('Headset', 35);
INSERT INTO inventory_log (product_id, change, changed_at) VALUES
  (1, -3,  '2026-03-10 09:00:00'),
  (2, +20, '2026-03-10 10:00:00'),
  (3, -5,  '2026-03-11 14:30:00'),
  (1, -2,  '2026-03-12 16:00:00');
  1. Write an AFTER INSERT ON inventory_log trigger that adds NEW.change to products.stock_count for the matching product
  2. Test it: insert a new row (e.g. product 1, change -5) and check that the Laptop stock dropped accordingly
  3. Write a BEFORE INSERT ON inventory_log validation trigger that calls RAISE(ABORT, 'change cannot be zero') when NEW.change = 0
  4. Test the validation: try inserting a row with change = 0 — does it get rejected?
  5. Bonus: extend the AFTER trigger to also log a message into a new alerts table whenever stock_count drops below 10

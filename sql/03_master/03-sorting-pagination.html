Topic 3

ORDER BY, LIMIT, and OFFSET

Control row order and return only the page you need.

Source table

We will use this movies table for every query below.

id title genre year rating
1 Metro Rush Action 2005 7.4
2 Quiet Lake Drama 2012 8.1
3 Night Pulse Horror 2018 6.8
4 Skyline Code Action 2021 7.4
5 Tiny Planet Family 2010 7.1
6 Silver Track Thriller 2016 7.4

Sort with one column

Start simple: sort by just one column before combining two columns.

Query 1
-- rating high to low
SELECT title, year, rating FROM movies
ORDER BY rating DESC;
title year rating
Quiet Lake 2012 8.1
Skyline Code 2021 7.4
Silver Track 2016 7.4
Metro Rush 2005 7.4
Tiny Planet 2010 7.1
Night Pulse 2018 6.8

Sort with multiple columns

The second column matters when rows need an extra ordering rule.

Query 2
-- rating high to low, then newest first
SELECT title, year, rating FROM movies
ORDER BY rating DESC, year ASC;
title year rating
Quiet Lake 2012 8.1
Metro Rush 2005 7.4
Silver Track 2016 7.4
Skyline Code 2021 7.4
Tiny Planet 2010 7.1
Night Pulse 2018 6.8
Query 3
-- genre A-Z, then title A-Z
SELECT title, genre FROM movies
ORDER BY genre ASC, title ASC;
title genre
Metro Rush Action
Skyline Code Action
Quiet Lake Drama
Tiny Planet Family
Night Pulse Horror
Silver Track Thriller

Pagination with LIMIT + OFFSET

Pagination only makes sense if the rows are in a fixed order first.

Query 4
-- first 3 rows
SELECT id, title FROM movies
ORDER BY id
LIMIT 3;
id title
1 Metro Rush
2 Quiet Lake
3 Night Pulse
Query 5
-- next page (skip first 3, take 3)
SELECT id, title FROM movies
ORDER BY id
LIMIT 3 OFFSET 3;
id title
4 Skyline Code
5 Tiny Planet
6 Silver Track
For stable paging, always combine LIMIT/OFFSET with ORDER BY

Why is it more stable? Because ORDER BY gives the rows one fixed order before SQLite cuts them into pages. Without ORDER BY, page 1 and page 2 may change order between runs, so the same row can move between pages.

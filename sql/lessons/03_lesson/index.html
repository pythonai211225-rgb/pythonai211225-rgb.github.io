Solution — Auto Increment

Step 1 — Create table with AUTOINCREMENT

CREATE TABLE support_tickets ( id INTEGER PRIMARY KEY AUTOINCREMENT, customer_name TEXT NOT NULL, priority TEXT, issue_score INTEGER );

Step 2 — Insert 3 rows without specifying id

INSERT INTO support_tickets (customer_name, priority, issue_score) VALUES ('Dana', 'high', 95), ('Omar', 'medium', 74), ('Noa', 'high', 88);

-- delete id = 3 (Noa) DELETE FROM support_tickets WHERE id = 3; -- insert a new row INSERT INTO support_tickets (customer_name, priority, issue_score) VALUES ('Liam', 'low', 60); SELECT id, customer_name FROM support_tickets;

id 3 was NOT reused — AUTOINCREMENT always moves forward. With plain INTEGER PRIMARY KEY, id 3 might have been reused.

Solution — Foreign Key

Step 1 — Create both tables

PRAGMA foreign_keys = ON; -- must be ON in SQLite for FK rules to be enforced CREATE TABLE projects ( id INTEGER PRIMARY KEY, project_name TEXT NOT NULL ); CREATE TABLE tasks ( id INTEGER PRIMARY KEY, task_name TEXT NOT NULL, project_id INTEGER NOT NULL, FOREIGN KEY (project_id) REFERENCES projects(id) );

Step 2 — Insert rows (parent table first)

-- insert into parent first! INSERT INTO projects (id, project_name) VALUES (1, 'Mobile App'), (2, 'Website Redesign'), (3, 'Data Migration'); -- then insert into child table INSERT INTO tasks (id, task_name, project_id) VALUES (1, 'Design Login Screen', 1), (2, 'Build Landing Page', 2), (3, 'Clean Old Records', 3), (4, 'Optimize Homepage Images', 2), (5, 'Validate Data Checksums', 3);

Step 3 — Find all tasks in project_id = 2

SELECT * FROM tasks WHERE project_id = 2;
Result:
idtask_nameproject_id
2Build Landing Page2
4Optimize Homepage Images2

Step 4 (Challenge) — JOIN to find by project name

SELECT tasks.task_name FROM tasks JOIN projects ON tasks.project_id = projects.id WHERE projects.project_name = 'Website Redesign';
Result:
task_name
Build Landing Page
Optimize Homepage Images

JOIN connects the two tables on the matching column. The FK ensures that every project_id in tasks points to a real row in projects.

Solution — WHERE + Logic

Setup — create and populate the movies table

CREATE TABLE movies ( id INTEGER PRIMARY KEY, title TEXT NOT NULL, genre TEXT NOT NULL, year INTEGER NOT NULL, rating REAL, minutes INTEGER ); INSERT INTO movies (id, title, genre, year, rating, minutes) VALUES (1, 'Metro Rush', 'Action', 2005, 7.9, 118), (2, 'Quiet Lake', 'Drama', 2012, 8.1, 124), (3, 'Night Pulse', 'Horror', 2018, 6.8, 95), (4, 'Skyline Code', 'Action', 2021, 7.4, 109), (5, 'Tiny Planet', 'Family', 2010, 7.1, 86);

Query 1 — After 2010 and rating >= 7.0

SELECT title, year, rating FROM movies WHERE year > 2010 AND rating >= 7.0;
Result:
titleyearrating
Quiet Lake20128.1
Skyline Code20217.4

Night Pulse (2018) is excluded because its rating 6.8 is below 7.0. Tiny Planet is excluded because year 2010 is NOT greater than 2010.

Query 2 — Action or Horror, shorter than 110 minutes

SELECT title, genre, minutes FROM movies WHERE (genre = 'Action' OR genre = 'Horror') AND minutes < 110;
Result:
titlegenreminutes
Night PulseHorror95
Skyline CodeAction109

Metro Rush (Action, 118 min) is excluded because 118 is not less than 110. Parentheses are important — without them AND would bind tighter than OR and the result would be wrong.

Query 3 — Non-Drama with rating above 7.3

SELECT title, genre, rating FROM movies WHERE NOT genre = 'Drama' AND rating > 7.3;
Result:
titlegenrerating
Metro RushAction7.9
Skyline CodeAction7.4

Query 4 — Rating 7–8.9 OR year after 2015

SELECT title, year, rating FROM movies WHERE (rating > 7 AND rating < 8.9) OR year > 2015;
Result:
titleyearrating
Metro Rush20057.9
Quiet Lake20128.1
Night Pulse20186.8
Skyline Code20217.4
Tiny Planet20107.1

All 5 rows match. Night Pulse (6.8) fails the rating condition but passes year > 2015 (2018). Quiet Lake (8.1) passes rating condition (8.1 < 8.9).

Solution — LIKE / IN / NULL

Setup — create and populate the products table

CREATE TABLE products ( id INTEGER PRIMARY KEY, product_name TEXT NOT NULL, category TEXT NOT NULL, price REAL NOT NULL, stock INTEGER ); INSERT INTO products VALUES (1, 'Mouse Pro', 'Accessories', 49.0, 20), (2, 'Monitor 27', 'Displays', 289.0, 5), (3, 'Keyboard Lite', 'Accessories', 39.0, NULL), (4, 'Dock Station', 'Accessories', 129.0, 8), (5, 'Mini PC', 'Computers', 699.0, 3);

Query 1 — Names ending in "Pro" (LIKE)

SELECT product_name FROM products WHERE product_name LIKE '%Pro';
Result:
product_name
Mouse Pro

Query 2 — Price between 30 and 150 (BETWEEN)

SELECT product_name, price FROM products WHERE price BETWEEN 30 AND 150;
Result:
product_nameprice
Mouse Pro49.0
Keyboard Lite39.0
Dock Station129.0

BETWEEN is inclusive — both 30 and 150 would be included if they existed.

Query 3 — Computers or Displays (IN)

SELECT product_name, category FROM products WHERE category IN ('Computers', 'Displays');
Result:

Monitor 27Displays

Query 4 — Known stock (IS NOT NULL)

SELECT product_name, stock FROM products WHERE stock IS NOT NULL;
Result:

Monitor 275
Dock Station8
Mini PC3

Keyboard Lite is excluded — its stock is NULL (unknown). You cannot use stock != NULL; you must use IS NOT NULL.

Bonus — GLOB: starts with "M" and has a space

SELECT product_name FROM products WHERE product_name GLOB 'M* *';
Result:
product_name
Mouse Pro
Monitor 27
Mini PC

GLOB is case-sensitive (unlike LIKE). 'M* *' means: starts with capital M, then anything, then a space, then anything.

Solution — ORDER BY + LIMIT

Setup — create and populate the 6-row movies table

CREATE TABLE movies ( id INTEGER PRIMARY KEY, title TEXT NOT NULL, genre TEXT NOT NULL, year INTEGER NOT NULL, rating REAL ); INSERT INTO movies (id, title, genre, year, rating) VALUES (1, 'Metro Rush', 'Action', 2005, 7.4), (2, 'Quiet Lake', 'Drama', 2012, 8.1), (3, 'Night Pulse', 'Horror', 2018, 6.8), (4, 'Skyline Code', 'Action', 2021, 7.4), (5, 'Tiny Planet', 'Family', 2010, 7.1), (6, 'Silver Track', 'Thriller', 2016, 7.4);

Query 1 — Top 2 highest-rated

SELECT title, rating FROM movies ORDER BY rating DESC LIMIT 2;
Result:
titlerating
Quiet Lake8.1
Metro Rush7.4

Three movies share 7.4 — SQLite picks by insertion order when ratings are equal. To make it fully deterministic, add a tiebreaker: ORDER BY rating DESC, id ASC.

Query 2 — Genre A→Z, then rating Z→A

SELECT title, genre, rating FROM movies ORDER BY genre ASC, rating DESC;
Result:
titlegenrerating
Metro RushAction7.4
Skyline CodeAction7.4
Quiet LakeDrama8.1
Tiny PlanetFamily7.1
Night PulseHorror6.8
Silver TrackThriller7.4

Query 3 — First 4 movies by year ascending

SELECT title, year FROM movies ORDER BY year ASC LIMIT 4;
Result:
titleyear
Metro Rush2005
Tiny Planet2010
Quiet Lake2012
Silver Track2016

Query 4 — Page 2 (page size = 2)

-- page 1 → LIMIT 2 OFFSET 0 (rows 1-2) -- page 2 → LIMIT 2 OFFSET 2 (rows 3-4) SELECT id, title FROM movies ORDER BY id LIMIT 2 OFFSET 2;
Result:
idtitle
3Night Pulse
4Skyline Code

OFFSET 2 skips the first 2 rows. Always combine LIMIT/OFFSET with ORDER BY for stable pagination — without ORDER BY the rows may appear in a different order on each run.

The db.py Library

Instead of opening and closing a connection in every file, we wrote one db.py with three reusable functions. Every other Python file simply imports from it.

import sqlite3 DB_PATH = "my_sql.db" def connect(): conn = sqlite3.connect(DB_PATH) conn.row_factory = sqlite3.Row return conn def run_query_select(sql, params=()): conn = connect() try: cursor = conn.cursor() cursor.execute(sql, params) rows = cursor.fetchall() return [dict(row) for row in rows] finally: conn.close() def run_update_query(sql, params=()): conn = connect() try: cursor = conn.cursor() cursor.execute(sql, params) conn.commit() return cursor.lastrowid finally: conn.close()
connect()
Opens (or creates) the .db file. Sets row_factory = sqlite3.Row so you can read columns by name: row["name"] instead of row[0].
run_query_select()
Use for SELECT. Returns a list of dicts — easy to loop and print. Closes the connection automatically in finally.
run_update_query()
Use for INSERT / UPDATE / DELETE / CREATE. Calls commit() to save changes to disk. Returns lastrowid of any INSERT.

How to Import

from db import run_query_select, run_update_query
Both db.py and main.py must be in the same folder. Python finds db.py by its filename, no path needed.

SQL Injection — The Attack

If you build a SQL string by pasting user input directly into it, an attacker can type SQL code as their "answer" and break your query logic entirely.

❌ Dangerous — string concatenation:

# user types: ' OR '1'='1 name = input("Enter name: ") sql = "SELECT * FROM users WHERE name = '" + name + "'" # becomes: SELECT * FROM users WHERE name = '' OR '1'='1' # the WHERE is always TRUE — attacker sees every row!
The problem: user input merges into the SQL text. The ' OR '1'='1 trick makes the condition always true — all rows are returned.

The Fix — ? Placeholders

Replace string building with ? markers and pass the values as a separate tuple. SQLite handles all escaping for you — the value can never become SQL code.

✅ Safe — ? placeholders:

name = input("Enter name: ") rows = run_query_select( "SELECT * FROM users WHERE name = ?", (name,) # trailing comma makes it a tuple! )
How it works: the SQL text and the value travel separately to SQLite. The ? is never interpreted as SQL — it is always just data. Injection is impossible.

Multiple Parameters

# One ? per value, filled left-to-right run_update_query( "INSERT INTO students (name, grade) VALUES (?, ?)", ("Alice", 90) )
Always use a tuple — even for a single value write (value,) with the trailing comma, otherwise Python won't treat it as a tuple.

INNER JOIN in Python — inner_ex.py

We took the INNER JOIN query from Lesson 5 and ran it from Python. Two related tables, one FOREIGN KEY, one join.

Step 1 — Enable Foreign Keys

run_update_query("PRAGMA foreign_keys = ON;")
Required: SQLite does not enforce FOREIGN KEY constraints by default. Run PRAGMA foreign_keys = ON before any INSERT that depends on a FK.

Step 2 — Create Tables

run_update_query("""DROP TABLE IF EXISTS employees;""") run_update_query("""DROP TABLE IF EXISTS departments;""") run_update_query("""CREATE TABLE departments ( id INTEGER PRIMARY KEY, dept_name TEXT NOT NULL, budget REAL NOT NULL );""") run_update_query("""CREATE TABLE employees ( id INTEGER PRIMARY KEY, name TEXT NOT NULL, dept_id INTEGER, salary REAL NOT NULL, FOREIGN KEY (dept_id) REFERENCES departments(id) );""")

Step 3 — Insert Data

run_update_query("INSERT INTO departments VALUES (?, ?, ?)", (1, "Engineering", 150000)) run_update_query("INSERT INTO departments VALUES (?, ?, ?)", (2, "Marketing", 80000)) run_update_query("INSERT INTO departments VALUES (?, ?, ?)", (3, "Sales", 60000)) run_update_query("INSERT INTO departments VALUES (?, ?, ?)", (4, "HR", 45000)) run_update_query("INSERT INTO employees VALUES (?, ?, ?, ?)", (1, "Dana", 1, 9000)) run_update_query("INSERT INTO employees VALUES (?, ?, ?, ?)", (2, "Omar", 2, 7200)) run_update_query("INSERT INTO employees VALUES (?, ?, ?, ?)", (3, "Noa", 1, 6800)) run_update_query("INSERT INTO employees VALUES (?, ?, ?, ?)", (4, "Liam", 3, 5500)) run_update_query("INSERT INTO employees VALUES (?, ?, ?, ?)", (5, "Rina", None, 6000)) run_update_query("INSERT INTO employees VALUES (?, ?, ?, ?)", (6, "Kai", 2, 7000))
Rina has dept_id = None (NULL) — she has no department and will not appear in INNER JOIN results.

Step 4 — Run INNER JOIN

result = run_query_select(""" SELECT e.name, d.dept_name, d.budget FROM employees e INNER JOIN departments d ON e.dept_id = d.id """) for e in result: print(e)

Making It Interactive

We extended inner_ex.py with four features. Python's input() pauses and waits for the user to type.

①
Input employee details → INSERT
Prompt for id, name, dept_id, salary, then insert the new row.
②
Search for employee by ID
Prompt for an id and SELECT that specific row. Print "not found" if nothing comes back.
③
try / except
Wrap the INSERT in a try block. If it fails (e.g. FK violation, duplicate id) the except prints a message instead of crashing.
④
PRAGMA foreign_keys = ON
Without this, SQLite silently accepts invalid dept_id values. With it, the INSERT raises an error that except catches.

① Input → INSERT

id = int(input("What is your id? ")) name = input("What is your name? ") dept_id = input("What is your dept_id?") salary = input("What is your salary? ") run_update_query( "INSERT INTO employees VALUES (?, ?, ?, ?)", (id, name, dept_id, salary) )
input() always returns a string. We wrap id in int() so it is stored as an integer, not text.

② Search by ID

id = int(input("Which employee id to search? ")) result = run_query_select( "SELECT * FROM employees WHERE id = ?", (id,) ) if result: print(result[0]) else: print("No employee found with that id.")
run_query_select always returns a list. If nothing matches, the list is empty — so if result is False.

③ try / except

id = int(input("What is your id? ")) name = input("What is your name? ") dept_id = input("What is your dept_id?") salary = input("What is your salary? ") try: run_update_query( "INSERT INTO employees VALUES (?, ?, ?, ?)", (id, name, dept_id, salary) ) print("Employee added successfully!") except Exception as e: print("Failed to insert:", e)
When does except run? If you enter a dept_id that doesn't exist in departments while PRAGMA is ON, SQLite raises FOREIGN KEY constraint failed. The except block catches it and prints the message — the program doesn't crash.

④ PRAGMA foreign_keys = ON

run_update_query("PRAGMA foreign_keys = ON;")
❌ Without PRAGMA: inserting an employee with dept_id = 99 (no such department) silently succeeds — bad data in, no error out.
✅ With PRAGMA: SQLite checks the foreign key and raises FOREIGN KEY constraint failed — the try/except catches it cleanly.
Run this line before any INSERT that depends on a FK. Put it near the top of inner_ex.py, right after your imports.

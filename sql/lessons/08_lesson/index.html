1NF — First Normal Form

The foundation of normalization. A table is in 1NF when every column holds atomic (single) values, there are no repeating groups, and every row is unique.

①
Atomic values — one value per cell, never a comma list or JSON blob.
②
No repeating groups — don't add columns phone1, phone2, phone3. Use a separate table instead.
③
Every row is unique — identified by a PRIMARY KEY.
④
Consistent column types — all values in a column are the same kind (all dates, all text, etc.).
✏

Q1 — All 1NF violations:

Violation 1 — Non-atomic values: The emails column stores "ana@a.com, ana@b.com" in a single cell — two values separated by a comma. A cell must hold exactly one value.
Violation 2 — Repeating groups: tag1, tag2, tag3 are the same concept repeated across columns. If a contact gets a 4th tag, you'd have to ALTER TABLE and add tag4 — that's a design smell.
Violation 3 — NULL waste: Ana's tag3 and João's tag2/tag3 are all NULL — wasted space for columns that don't apply to those rows.

✅ Good — Conforms to 1NF (exercise solution)

Complete Solution — All 6 Questions

Q3 — CREATE TABLE statements:

CREATE TABLE contacts ( id INTEGER PRIMARY KEY AUTOINCREMENT, full_name TEXT NOT NULL ); CREATE TABLE contact_emails ( id INTEGER PRIMARY KEY AUTOINCREMENT, contact_id INTEGER NOT NULL, email TEXT NOT NULL, FOREIGN KEY (contact_id) REFERENCES contacts(id) ON DELETE CASCADE ); CREATE TABLE contact_tags ( contact_id INTEGER NOT NULL, tag TEXT NOT NULL, PRIMARY KEY (contact_id, tag), FOREIGN KEY (contact_id) REFERENCES contacts(id) ON DELETE CASCADE );

Q4 — INSERT data from original table:

INSERT INTO contacts (full_name) VALUES ('Ana Silva'), ('João Souza'); INSERT INTO contact_emails (contact_id, email) VALUES (1, 'ana@a.com'), -- Ana has two emails (1, 'ana@b.com'), (2, 'joao@c.com'); INSERT INTO contact_tags (contact_id, tag) VALUES (1, 'VIP'), (1, 'Newsletter'), (2, 'Newsletter');

SELECT c.id, c.full_name FROM contacts c JOIN contact_tags ct ON ct.contact_id = c.id WHERE ct.tag = 'Newsletter';

2NF — Second Normal Form

A table is in 2NF when it is already in 1NF and every non-key column depends on the entire primary key — not just part of it.

Only matters with composite PKs. A table with a single-column PK is automatically in 2NF (assuming it's already in 1NF).

Partial Dependency

A partial dependency happens when a non-key column depends on only part of a composite primary key.

Example — composite PK is (student_id, course_id):

student_id + course_id
composite PK
→
grade
✅ full dependency
≠
student_name
❌ partial (only student_id)

3NF — Third Normal Form

A table is in 3NF when it is in 2NF and every non-key column depends only on the primary key — never on another non-key column.

Transitive Dependency

A transitive dependency exists when a non-key column determines another non-key column, creating an indirect path to the PK:

PK → column A → column B
employee_id
PK
→
dept_id
non-key
→
dept_name
❌ transitive

dept_name depends on dept_id, not directly on employee_id — that's a transitive dependency.

Why it's a problem: If you rename a department, you have to update every employee row in that department. Miss one row → inconsistent data.

Ref Notation

The Ref: keyword defines relationships. The symbol shows the cardinality:

SymbolMeaningExample
>Many-to-One (FK → PK)orders.customer_id > customers.id
<One-to-Manycustomers.id < orders.customer_id
-One-to-Oneusers.id - profiles.user_id
<>Many-to-Many(use a junction table instead)

dbdiagram Syntax

Here is the dbdiagram syntax for a simple orders schema. Type this at dbdiagram.io and the diagram appears immediately:

// Customers table Table customers { id integer [pk, increment] full_name varchar [not null] email varchar [unique, not null] phone varchar } // Orders table Table orders { id integer [pk, increment] customer_id integer [not null, ref: > customers.id] ordered_at datetime total decimal } // Products table Table products { id integer [pk, increment] name varchar [not null] price decimal [not null] } // Junction table — order_items Table order_items { order_id integer [pk, ref: > orders.id] product_id integer [pk, ref: > products.id] qty integer [not null] unit_price decimal [not null] // price snapshot at order time } // Explicit relationships (optional — Ref: inside table works too) Ref: orders.customer_id > customers.id Ref: order_items.order_id > orders.id Ref: order_items.product_id > products.id

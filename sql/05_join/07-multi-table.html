Joining 3+ Tables

Chain JOIN clauses one after another. Each JOIN adds another table. Order matters for readability; the SQL engine optimizes internally.

Example 1 — 3-Table JOIN: projects + employees + departments

Query
SELECT
  p.project_name  AS project,
  e.name          AS lead,
  d.dept_name     AS department,
  d.budget        AS dept_budget
FROM   projects p
JOIN   employees   e ON p.lead_id = e.id
JOIN   departments d ON e.dept_id = d.id;

Each JOIN adds a new table. The engine first joins projects → employees, then joins the result → departments.

Example 2 — Mix of LEFT JOINs: all employees + optional dept + optional project

Query
SELECT
  e.name         AS employee,
  d.dept_name    AS department,
  p.project_name AS project_led
FROM   employees   e
LEFT JOIN departments d ON e.dept_id = d.id
LEFT JOIN projects    p ON p.lead_id = e.id;

Example 3 — Aggregation: total spend per customer

Query
SELECT
  c.name                              AS customer,
  SUM(p.price * o.quantity)           AS total_spent
FROM   orders    o
JOIN   customers c ON o.customer_id = c.id
JOIN   products  p ON o.product_id  = p.id
GROUP BY c.id, c.name
ORDER BY total_spent DESC;

Multiply price × quantity per row, then SUM() aggregates all rows per customer. ORDER BY total_spent DESC puts the top spender first.

Order of JOINs — Tips

Good patterns

  • Start with the "anchor" table (most rows, central FK)
  • Use INNER JOIN for required relationships
  • Use LEFT JOIN for optional relationships
  • Put conditions in ON, not WHERE, for LEFT JOINs

Common mistakes

  • Forgetting an ON clause (creates accidental CROSS JOIN)
  • Using the wrong key column in ON
  • Moving a LEFT JOIN's ON condition to WHERE (turns it into INNER JOIN)
  • 1-to-many JOINs producing duplicate rows

The SQL engine reorders JOINs internally for performance — the logical order you write only matters for readability and correctness of LEFT/RIGHT semantics.

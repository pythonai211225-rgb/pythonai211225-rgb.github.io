Why not just files or Excel?

Files and spreadsheets feel simple — and they are, until they break. Here's exactly where they fall apart, and why databases were invented to fix those problems

📊 Excel has a 1,048,576 row limit 💥 CSVs corrupt under concurrent writes 🔒 Databases enforce rules automatically
A real scenario

Imagine you're building a school system

Your school has 800 students, 60 teachers, and 200 courses. You need to track enrolments, grades, attendance, and payments. A friend suggests "just use a spreadsheet." Here's what happens next.

⚠️ Week 1 — It seems fine

You have one students.csv file. You add rows manually. Life is good. The file is 12 KB.

⚠️ Month 3 — Cracks appear

The admin and the finance team both have the file open at the same time. Finance saves their version. Admin's changes are silently lost. You now have two conflicting copies and no idea which is correct.

💥 Year 1 — It breaks completely

The file is 80 MB. Excel freezes for 30 seconds every time you filter. A formula references the wrong column and miscalculates 600 student grades. There's no record of who changed what or when. You have no way to roll back.

Side by side

Files & Excel vs a database

📄 Files / Excel

  • One person can safely edit at a time — others must wait or risk data loss
  • No rules — you can type "banana" in an age column and nothing stops you
  • Searches require reading every row from top to bottom (slow)
  • Duplicates accumulate silently — same student registered twice
  • No history — you can't see who changed row 472 last Tuesday
  • Relationships between sheets break when rows move or are deleted
  • Excel max: ~1 million rows. Typical app: hundreds of millions

🗄️ SQL Database

  • Thousands of users reading and writing at the exact same time — safely
  • Constraints enforce rules — "banana" in an INTEGER column is rejected automatically
  • Indexes make queries fast — finding one row in 100 million takes milliseconds
  • UNIQUE constraints prevent duplicate entries at the database level
  • Transaction logs record every change with a timestamp and author
  • Foreign keys keep relationships consistent even when data changes
  • No practical row limit — PostgreSQL tables have been tested to tens of billions of rows
The biggest problem: concurrency

What happens when two people edit at once

This is called a race condition — two writers overlap and one silently destroys the other's work

❌ With a CSV file

  1. 10:00:01 — Alice reads the file (500 students)
  2. 10:00:02 — Bob reads the file (500 students)
  3. 10:00:05 — Alice adds student #501, saves (501 students)
  4. 10:00:06 — Bob adds student #502 to his copy, saves (501 students)
  5. Alice's student #501 is permanently gone. No error. No warning.

✓ With a database

  1. 10:00:01 — Alice sends INSERT command
  2. 10:00:01 — Bob sends INSERT command simultaneously
  3. 10:00:01 — Database queues both, runs them in order
  4. 10:00:01 — Both inserts succeed. Table now has 502 students
  5. No data lost. Handled automatically. Neither Alice nor Bob did anything special.
Data integrity

Databases enforce rules so your code doesn't have to

Without a database, every app that writes data must manually check for bad values. With a database, you declare the rules once in the table definition — and the database rejects bad data for every app, forever

-- Declare the rules once...
CREATE TABLE students (
    id      INTEGER PRIMARY KEY AUTOINCREMENT,
    name    TEXT    NOT NULL,
    email   TEXT    NOT NULL UNIQUE,   -- no duplicate emails ever
    age     INTEGER CHECK(age BETWEEN 5 AND 120),
    grade   REAL    CHECK(grade BETWEEN 0.0 AND 100.0)
);

-- These will be rejected automatically:
INSERT INTO students (name, email, age) VALUES ('Bob', 'bob@x.com', -5);   -- age -5 fails CHECK
INSERT INTO students (name, email)         VALUES ('Eve',  'bob@x.com');       -- duplicate email fails UNIQUE
INSERT INTO students (email)               VALUES ('anon@x.com');             -- missing name fails NOT NULL
The database said no — none of those inserts went through. The data stayed clean without a single line of validation code in your app
To be fair

When files actually are the right choice

Databases aren't always the answer. Files are perfectly fine when:

  • ✓ Only one person ever reads or writes the data
  • ✓ The data is static config — read once at startup, never changed at runtime
  • ✓ You have fewer than a few thousand rows and no complex queries
  • ✓ You're prototyping something quick and will move to a proper database later
Fun fact: SQLite itself stores its database in a single file on disk — so even when you "use a database," you still have a file. The difference is that the database engine manages that file safely for you

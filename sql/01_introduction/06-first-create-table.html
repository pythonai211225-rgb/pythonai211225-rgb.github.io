Topic 6 of 6 — The practical bit

Your first CREATE TABLE

A table is the fundamental unit of a relational database. Every piece of data you store lives in a row inside a table. Here's how to design one properly — with types, constraints, and real sample data

5 data types 6 constraints 3 worked examples
The structure

Anatomy of CREATE TABLE

Every table definition follows the same pattern: give it a name, then list each column with its data type and any rules (constraints) that apply to it

CREATE TABLE table_name (
    column_name   DATA_TYPE   CONSTRAINT,
    another_col   DATA_TYPE,
    ...
);
IF NOT EXISTS — always write CREATE TABLE IF NOT EXISTS table_name in real scripts. Without it, re-running the script throws an error because the table already exists
Step 1

Data types — what kind of value does each column hold?

SQLite has five storage classes. Everything you store ends up as one of these

INTEGER

Whole numbers only — no decimals. Stored in 1 to 8 bytes depending on the size of the value. Use for IDs, counts, ages, quantities

→ 1, 42, -7, 1000000

REAL

Numbers with decimals. Stored as a standard 64-bit float — the same type your calculator uses. Use for prices, ratings, coordinates, measurements

→ 3.14, 99.99, -0.5

TEXT

Any string of characters — names, emails, URLs, JSON, and also dates (SQLite has no native date type). Stored as UTF-8. No length limit

→ 'Alice', '2024-03-15 08:22:11'

BLOB

Binary data — images, audio, files, raw bytes. Stored exactly as provided, byte for byte. Usually you'd store a path to a file rather than the file itself

→ X'89504e47' (PNG header bytes)

NULL

Means "no value" — not zero, not an empty string. NULL is the absence of data. Every column accepts NULL by default unless you add NOT NULL

→ unknown, not provided, optional

BOOLEAN

SQLite has no separate boolean type. True/false values are stored as INTEGER 1 or 0. Declare the column as INTEGER and add a CHECK constraint to enforce it

→ 1 = true, 0 = false

Dates in SQLite: store them as TEXT in ISO format 'YYYY-MM-DD HH:MM:SS'. This format sorts alphabetically in the correct chronological order, and SQLite's built-in datetime() and strftime() functions work with it natively
Step 2

Constraints — rules that protect your data

A constraint is a rule the database enforces on every INSERT and UPDATE. Declare it once, and it applies everywhere — no matter which app, script, or person writes data

  • PK

    PRIMARY KEY — the row's unique identity

    Every table should have one. Uniquely identifies each row. Combined with INTEGER and AUTOINCREMENT, the database assigns the next available number automatically — you never set it yourself

    id  INTEGER  PRIMARY KEY AUTOINCREMENT
  • NN

    NOT NULL — this column must always have a value

    Without NOT NULL, a column is optional — you can skip it on INSERT and it becomes NULL. Add NOT NULL to make it required. If you try to insert a row without providing a NOT NULL column, the database rejects it with an error

    name  TEXT  NOT NULL
  • UQ

    UNIQUE — no two rows may share this value

    Useful for emails, usernames, phone numbers, product codes — anything that must identify exactly one row. Unlike PRIMARY KEY, a table can have multiple UNIQUE columns. NULL values are exempt — two rows can both have NULL in a UNIQUE column

    email  TEXT  NOT NULL UNIQUE
  • DF

    DEFAULT — fill in a value when none is provided

    If you don't provide this column on INSERT, the database fills in the default. Can be a fixed value like 'pending' or 0, or a function call like (datetime('now')) which inserts the current timestamp

    status      TEXT     NOT NULL DEFAULT 'pending'
created_at  TEXT     NOT NULL DEFAULT (datetime('now'))
  • CK

    CHECK — a custom rule expressed as an expression

    The most flexible constraint. Write any boolean expression — if it evaluates to false, the row is rejected. Use it to enforce ranges, allowed values, or relationships between columns in the same row

    age         INTEGER  CHECK(age BETWEEN 0 AND 130)
rating      REAL     CHECK(rating BETWEEN 1.0 AND 5.0)
difficulty  TEXT     CHECK(difficulty IN ('easy','normal','hard'))
🧑 Worked example 1

A user accounts table

CREATE TABLE IF NOT EXISTS users (

    -- Auto-assigned number. You never set this yourself
    id          INTEGER  PRIMARY KEY AUTOINCREMENT,

    -- Display name: required, and each username must be different
    username    TEXT     NOT NULL UNIQUE,

    -- Email: required and must be unique — used for login
    email       TEXT     NOT NULL UNIQUE,

    -- Age is optional (NULL = not provided), but if given must be sensible
    age         INTEGER  CHECK(age BETWEEN 0 AND 130),

    -- 1 = active, 0 = banned/deactivated. Defaults to active
    is_active   INTEGER  NOT NULL DEFAULT 1  CHECK(is_active IN (0, 1)),

    -- ISO timestamp: 'YYYY-MM-DD HH:MM:SS'. Auto-filled by the database
    created_at  TEXT     NOT NULL DEFAULT (datetime('now'))
);
what that table looks like with real data
id int username text email text age int is_active int created_at text
1'alice99''alice@example.com'281'2024-03-15 08:22:11'
2'bob_dev''bob@gmail.com'341'2024-03-16 14:05:33'
3'zara_x''zara@mail.com'220'2024-03-17 09:11:44'
4'ghost''ghost@dark.net'NULL1'2024-03-18 21:59:00'

Row 4: age is NULL — the user chose not to provide it, and that's fine because the column has no NOT NULL constraint. Row 3: is_active is 0 — this account is deactivated

🛒 Worked example 2

A product inventory table

CREATE TABLE IF NOT EXISTS products (
    id           INTEGER  PRIMARY KEY AUTOINCREMENT,
    name         TEXT     NOT NULL,

    -- Store price as integer cents to avoid floating-point rounding errors
    -- £29.99 → store 2999. Divide by 100 when displaying to the user
    price_pence  INTEGER  NOT NULL  CHECK(price_pence >= 0),

    -- Stock defaults to 0 (not yet stocked), can never go negative
    stock        INTEGER  NOT NULL DEFAULT 0  CHECK(stock >= 0),

    -- Enum-style: only these four category values are accepted
    category     TEXT     CHECK(category IN ('electronics','clothing','food','other')),

    -- Customer rating 1–5 stars. NULL means no reviews yet
    rating       REAL     CHECK(rating BETWEEN 1.0 AND 5.0)
);
sample data
id name text price_pence int stock int category text rating real
1'Wireless Mouse'299942'electronics'4.5
2'Blue T-Shirt'1499120'clothing'3.8
3'Fancy Coffee'8998'food'4.9
4'USB-C Hub'45990'electronics'NULL

Row 4: stock is 0 (out of stock) and rating is NULL (brand new, no reviews yet). Price 2999 = £29.99 — your app divides by 100 before displaying

🎮 Worked example 3

A game leaderboard table

CREATE TABLE IF NOT EXISTS high_scores (
    id           INTEGER  PRIMARY KEY AUTOINCREMENT,
    player_name  TEXT     NOT NULL,

    -- Score can never be negative — no such thing as a negative score
    score        INTEGER  NOT NULL  CHECK(score >= 0),

    -- Level must be at least 1. Defaults to 1 if not provided
    level        INTEGER  NOT NULL DEFAULT 1  CHECK(level >= 1),

    -- SQLite has no ENUM type — use CHECK with IN() to fake one
    difficulty   TEXT     NOT NULL DEFAULT 'normal'
                           CHECK(difficulty IN ('easy','normal','hard')),

    -- ISO timestamp. Sorts alphabetically = sorts chronologically
    played_at    TEXT     NOT NULL DEFAULT (datetime('now'))
);

Seeing constraints in action

What the database blocks automatically

Using the users table from example 1 — here are inserts the database will reject, and exactly why

-- ✗ REJECTED: age -5 fails CHECK(age BETWEEN 0 AND 130)
INSERT INTO users (username, email, age)
VALUES ('oops', 'oops@x.com', -5);

-- ✗ REJECTED: 'bob@gmail.com' already exists — fails UNIQUE
INSERT INTO users (username, email)
VALUES ('newbob', 'bob@gmail.com');

-- ✗ REJECTED: username is missing — fails NOT NULL
INSERT INTO users (email)
VALUES ('anon@x.com');

-- ✓ ACCEPTED: age not provided → NULL (no NOT NULL on age)
-- created_at not provided → database fills in datetime('now') automatically
INSERT INTO users (username, email)
VALUES ('newuser', 'new@x.com');
The database said no — three of those four inserts were rejected without a single line of validation code in your app. The data stays clean no matter how many apps write to the same database
Quick reference

Cheat sheet — patterns you'll use every time

-- Auto-incrementing primary key (every table needs this)
id          INTEGER  PRIMARY KEY AUTOINCREMENT

-- Required text
name        TEXT     NOT NULL

-- Required + no duplicates allowed
email       TEXT     NOT NULL UNIQUE

-- Optional (no constraints = nullable by default)
bio         TEXT

-- Default value when not provided
status      TEXT     NOT NULL DEFAULT 'active'

-- Validated range
age         INTEGER  CHECK(age BETWEEN 0 AND 130)

-- Enum-style (limited allowed values)
role        TEXT     CHECK(role IN ('admin','user','guest'))

-- Boolean (1 = true, 0 = false)
is_active   INTEGER  NOT NULL DEFAULT 1  CHECK(is_active IN (0,1))

-- Auto-filled timestamp
created_at  TEXT     NOT NULL DEFAULT (datetime('now'))

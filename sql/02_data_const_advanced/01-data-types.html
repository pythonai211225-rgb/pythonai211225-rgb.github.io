Type Affinity
& Strict Mode

SQLite's type system is unlike any other database. Before diving into individual types, understand the rules that govern how SQLite decides what to store — and how to switch on strict typing when you need it.

5 storage classes Dynamic typing by default STRICT mode available typeof() to inspect
The core difference

SQLite is dynamically typed — by default

In PostgreSQL or MySQL, the column type is a hard rule — insert the wrong type and you get an error. SQLite is different: the value itself carries its type, not the column. The column declaration is a hint called affinity.

MySQL / PostgreSQL

  • Column type is enforced strictly
  • Wrong type → immediate error
  • VARCHAR(50) rejects strings > 50 chars
  • No implicit conversion

SQLite (default)

  • Column type is a hint (affinity)
  • SQLite converts when possible
  • VARCHAR(999999) = just TEXT
  • typeof('42') can be 'integer'

Affinity rules — applied in order

How SQLite picks the affinity for a column

SQLite looks at the declared type name and applies these 5 rules in this exact order. First match wins:

RuleIf declared type contains…AffinityExamples
1 "INT" INTEGER INT, INTEGER, TINYINT, BIGINT, BOOLEAN
2 "CHAR" / "CLOB" / "TEXT" TEXT TEXT, VARCHAR(n), CHAR(n), NVARCHAR, CLOB
3 "BLOB" or empty NONE BLOB, (no type declared)
4 "REAL" / "FLOA" / "DOUB" REAL REAL, FLOAT, DOUBLE, DOUBLE PRECISION
5 anything else NUMERIC NUMERIC, DECIMAL, NUMBER, DATE, DATETIME
typeof() — always know what SQLite stored:
SELECT typeof(42) → 'integer'  |  SELECT typeof('hi') → 'text'  |  SELECT typeof(3.14) → 'real'  |  SELECT typeof(NULL) → 'null' 
-- See what affinity each column has
CREATE TABLE test (
  a INT,        -- INTEGER affinity (contains "INT")
  b VARCHAR,    -- TEXT affinity (contains "CHAR")
  c BLOB,       -- NONE affinity
  d FLOAT,      -- REAL affinity (contains "FLOA")
  e DECIMAL,    -- NUMERIC affinity (none of the above)
  f BOOLEAN,    -- INTEGER affinity! (contains "INT"... wait, no)
  -- BOOLEAN → rule 5 (catch-all) → NUMERIC affinity
  -- but values are still stored as 0/1 integers
);

-- Check what was actually stored
INSERT INTO test VALUES ('42', 100, 1, '3.14', '99', TRUE);
SELECT typeof(a), typeof(b), typeof(c),
       typeof(d), typeof(e), typeof(f) FROM test;
-- integer | integer | integer | real | integer | integer
SQLite 3.37+ feature

STRICT mode — opt-in hard typing

Added in SQLite 3.37.0 (2021). Append STRICT to your CREATE TABLE to turn on strict type enforcement — just like MySQL or PostgreSQL.

Regular table — flexible, permissive 
CREATE TABLE flexible (
  id   INTEGER,
  name TEXT
); -- default

-- ✓ Allowed — SQLite converts
INSERT INTO flexible
VALUES ('5', 42);
-- id='5'→integer, name=42→'42'
STRICT table — enforced like other DBs 
CREATE TABLE strict_t (
  id   INTEGER,
  name TEXT
) STRICT;

-- ✗ Fails — wrong type!
INSERT INTO strict_t
VALUES ('5', 42);
-- Error: cannot store TEXT in INTEGER
STRICT allowed types: INT, INTEGER, REAL, TEXT, BLOB, ANY. That's it — no FLOAT, no VARCHAR, no BOOLEAN. Use ANY for a column that truly accepts any type. 
-- STRICT mode with ANY column
CREATE TABLE events (
  id        INTEGER  PRIMARY KEY,
  name      TEXT     NOT NULL,
  payload   ANY,     -- accepts anything
  score     REAL,
  raw       BLOB
) STRICT;   -- type errors become hard errors

-- ✓ Fine — correct types
INSERT INTO events (id, name, score)
VALUES (1, 'login', 9.5);

-- ✗ Fails — score must be REAL
INSERT INTO events (id, name, score)
VALUES (2, 'logout', 'fast');
-- Error: cannot store TEXT in REAL column score
Storage Class 1 of 5 NONE affinity

∅ NULL

The absence of any value. Not zero. Not an empty string. Literally nothing. Every column is nullable by default unless you say NOT NULL.

0 bytes on disk Use IS NULL (not = NULL) NULL ≠ NULL COALESCE() for fallback
What NULL is

NULL means "we don't know" — not zero, not empty

Think of NULL as a question mark in your data. A user's phone number might be NULL because we don't have it yet. An order's delivery date might be NULL because it hasn't been delivered. NULL is the honest way to say "this information is missing."

idnamephone nullablebio nullable
1'Alice''+44 7911 123456'NULL
2'Bob'NULL'Developer at...'

Working with NULL

Always use IS NULL — never = NULL

CREATE TABLE users (
  id      INTEGER PRIMARY KEY,
  name    TEXT    NOT NULL,
  phone   TEXT,      -- nullable
  bio     TEXT       -- nullable
);

-- ✓ phone is optional — not supplied → NULL
INSERT INTO users (name) VALUES ('Alice');

-- ✓ Find users with no phone
SELECT name FROM users WHERE phone IS NULL;

-- ✗ WRONG — always returns nothing! NULL ≠ NULL
SELECT name FROM users WHERE phone = NULL;

-- ✓ Find users who DO have a phone
SELECT name FROM users WHERE phone IS NOT NULL;

-- COALESCE: use fallback when NULL
SELECT name, COALESCE(phone, 'No phone') AS contact
FROM users;
-- Alice | +44 7911 123456
-- Bob   | No phone
⚠️ NULL arithmetic: Any maths with NULL gives NULL back. 5 + NULL = NULL, COUNT(col) skips NULLs, SUM(col) ignores NULLs. Use COALESCE(col, 0) to treat NULL as zero.
IFNULL() — a handy shorthand: IFNULL(phone, 'unknown') is identical to COALESCE(phone, 'unknown') but only takes two arguments. Use COALESCE when you have multiple fallbacks.
NULL in aggregate functions

NULL is silently skipped in aggregates

CREATE TABLE scores (val INTEGER);
INSERT INTO scores VALUES (10), (20), (NULL), (30);

SELECT
  COUNT(*)         AS total_rows,   -- 4 (counts NULLs)
  COUNT(val)       AS non_null,     -- 3 (skips NULLs)
  SUM(val)         AS total,        -- 60 (skips NULLs)
  AVG(val)         AS average       -- 20 (60/3, not 60/4)
FROM scores;
Storage Class 2 of 5 INTEGER affinity

🔢 INTEGER

Whole numbers — stored in 1, 2, 3, 4, 6, or 8 bytes depending on the magnitude of the value. SQLite picks the smallest size automatically.

1–8 bytes (auto-sized) 64-bit signed max BOOLEAN → 0 or 1 DATE → text (recommended)
Declared types → INTEGER affinity

All these become INTEGER on disk

Any type name that contains "INT" gets INTEGER affinity (rule 1). Also any type that matches rule 5 (catch-all NUMERIC) — like BOOLEAN — stores whole number values as INTEGER:

INTEGER INT TINYINT SMALLINT MEDIUMINT BIGINT INT2 INT8 UNSIGNED BIG INT BOOLEAN

Special cases

BOOLEAN and DATE — both stored as INTEGER / TEXT

🎭 BOOLEAN is INTEGER: SQLite has no boolean type. TRUE stores as 1, FALSE stores as 0. You can query with WHERE active = TRUE — it works because SQLite treats it as = 1.
📅 DATE / DATETIME — store as TEXT: Declared DATE gets NUMERIC affinity (catch-all rule). Best practice: store dates as ISO 8601 TEXT '2024-12-25' — it sorts alphabetically = chronologically, and SQLite's built-in date functions understand it. 
CREATE TABLE employees (
  id         INTEGER PRIMARY KEY,
  name       TEXT    NOT NULL,
  salary     INTEGER NOT NULL,  -- store in pence/cents
  is_active  BOOLEAN DEFAULT TRUE, -- stored as 1
  hired_on   TEXT    NOT NULL   -- ISO date: '2024-03-01'
);

INSERT INTO employees (name, salary, hired_on)
VALUES
  ('Alice', 6500000, '2022-09-01'),
  ('Bob',   5800000, '2023-01-15');

-- BOOLEAN query works naturally
SELECT name FROM employees WHERE is_active = TRUE;

-- Date sort works because ISO format is alphsort-safe
SELECT name FROM employees ORDER BY hired_on ASC;
-- Alice | Bob

-- Salary in pence → display in £
SELECT name, salary / 100.0 AS salary_gbp FROM employees;
-- Alice | 65000.0
Storage Class 3 of 5 REAL affinity

📐 REAL

Floating-point numbers — always stored as an 8-byte IEEE 754 double-precision float. Great for measurements and scientific data, terrible for money.

Always 8 bytes 15–17 significant digits Never use for money!
Declared types → REAL affinity

All these become 8-byte floats on disk

Any type name containing "REAL", "FLOA", or "DOUB" gets REAL affinity (rule 4):

REAL FLOAT DOUBLE DOUBLE PRECISION
Example

Sensor readings — a perfect REAL use case

CREATE TABLE readings (
  id          INTEGER PRIMARY KEY,
  sensor      TEXT    NOT NULL,
  temperature REAL,             -- degrees Celsius
  humidity    FLOAT,            -- same as REAL
  pressure    DOUBLE,           -- same as REAL
  recorded    TEXT    NOT NULL
);

INSERT INTO readings (sensor,temperature,humidity,pressure,recorded) VALUES
  ('Room A',  21.5,  68.3,  1013.25, '2024-06-01 09:00'),
  ('Room B',  19.8,  72.1,  1011.80, '2024-06-01 09:00'),
  ('Outside', -3.2,  95.0,  995.60,  '2024-06-01 09:00');

-- Average of rooms above freezing
SELECT ROUND(AVG(temperature), 2) AS avg_temp
FROM   readings
WHERE  temperature > 0;
-- → 20.65
The money trap

⚠️ Never store money as REAL

IEEE 754 floats cannot represent most decimal fractions exactly. 0.1 + 0.2 in floating point is 0.30000000000000004 — which causes rounding errors in financial calculations.

❌ Wrong — REAL for money

  • 0.1 + 0.2 = 0.30000000000000004
  • Rounding errors accumulate
  • Totals can be off by a penny
  • Fails audit / compliance

✓ Right — INTEGER for money

  • Store in smallest unit (pence, cents)
  • 1499 = £14.99, never rounded
  • Divide by 100 only for display
  • Exact arithmetic, always
Storage Class 4 of 5 TEXT affinity

💬 TEXT

Character strings stored in UTF-8 (or UTF-16). Any length — SQLite completely ignores any length limit you write like VARCHAR(255).

UTF-8 by default No length limit enforced VARCHAR(n) = just TEXT LIKE / GLOB for pattern search
Declared types → TEXT affinity

All these become UTF-8 strings on disk

Any type containing "CHAR", "CLOB", or "TEXT" gets TEXT affinity (rule 2):

TEXT VARCHAR(n) CHAR(n) CHARACTER(n) NVARCHAR NCHAR CLOB NATIVE CHARACTER VARYING CHARACTER
VARCHAR(50) in SQLite does NOT limit to 50 characters. The number is completely ignored. You'd need a CHECK(length(col) <= 50) to actually enforce it.
Text search

LIKE and GLOB for pattern matching

CREATE TABLE articles (
  id        INTEGER PRIMARY KEY,
  title     VARCHAR NOT NULL,  -- TEXT, any length
  body      TEXT    NOT NULL,
  status    TEXT    NOT NULL,  -- 'draft'|'published'
  published TEXT              -- ISO date as TEXT
);

INSERT INTO articles (title, body, status, published) VALUES
  ('SQLite Guide',  'SQLite is great...', 'published', '2024-03-01'),
  ('SQL for Beginners', 'Let us learn SQL', 'published', '2024-04-10'),
  ('Draft Post',    'Work in progress...', 'draft', NULL);

-- LIKE: case-insensitive, % = any chars, _ = one char
SELECT title FROM articles WHERE title LIKE '%SQL%';
-- SQLite Guide | SQL for Beginners

-- GLOB: case-sensitive, * = any chars, ? = one char
SELECT title FROM articles WHERE title GLOB 'SQL*';
-- SQLite Guide

-- Dates stored as TEXT sort correctly!
SELECT title, published
FROM  articles
WHERE status = 'published'
ORDER BY published DESC;
-- SQL for Beginners | 2024-04-10
-- SQLite Guide      | 2024-03-01
Storage Class 5 of 5 NONE affinity

📦 BLOB

Binary Large Object — raw bytes stored exactly as given, with zero conversion, zero interpretation. The type you use when data is not text.

No conversion ever X'AABB...' hex literals Images, files, hashes No declared type = BLOB
Declared types → NONE affinity

BLOB and the "no type" column

BLOB (no type at all)
Fun fact: A column with no type declaration at all — just a column name — gets NONE affinity and behaves like BLOB. This is valid SQLite: CREATE TABLE t (a, b, c);
Example

Storing files, avatars, and cryptographic hashes

CREATE TABLE assets (
  id        INTEGER PRIMARY KEY,
  filename  TEXT    NOT NULL,
  mime_type TEXT    NOT NULL,
  data      BLOB,             -- raw file bytes
  sha256    BLOB              -- 32-byte hash
);

-- Insert with hex literal X'...'
INSERT INTO assets (filename, mime_type, data)
VALUES (
  'icon.png',
  'image/png',
  X'89504E470D0A1A0A0000000D49484452' -- PNG header bytes
);

-- Get file size in bytes
SELECT filename, LENGTH(data) AS bytes
FROM   assets;

-- Verify typeof is 'blob'
SELECT typeof(data) FROM assets;
-- → blob
💡 When to use BLOB vs file path: For small binaries (thumbnails <100KB, tokens, checksums) store as BLOB. For large files (videos, photos, documents) store the file path as TEXT and keep the file on disk — BLOBs in SQLite can make the database file very large and slow.
Affinity type NUMERIC affinity

🔀 NUMERIC

The catch-all affinity. When a declared type doesn't match any of the other four rules, SQLite gives it NUMERIC affinity — and tries to convert values into INTEGER or REAL whenever possible.

Converts TEXT → INTEGER if possible DECIMAL, NUMERIC types DATE/DATETIME affinity typeof() reveals the truth
Declared types → NUMERIC affinity

The catch-all — when rules 1–4 don't match

NUMERIC DECIMAL NUMBER DATE DATETIME anything else not covered
Conversion behaviour

NUMERIC tries hard to be a number

With NUMERIC affinity, SQLite attempts conversion in this order: first INTEGER, then REAL, then leaves as TEXT:

CREATE TABLE demo (val NUMERIC);

INSERT INTO demo VALUES
  ('42'),      -- text looks like integer → stored as INTEGER 42
  ('3.14'),    -- text looks like float   → stored as REAL 3.14
  ('hello'),   -- not a number at all     → stays as TEXT 'hello'
  (1.0),       -- float with no fraction  → stored as INTEGER 1
  (1.5),       -- float with fraction     → stays as REAL 1.5
  (NULL);      -- NULL stays NULL

SELECT val, typeof(val) FROM demo;
-- 42    | integer
-- 3.14  | real
-- hello | text
-- 1     | integer  ← 1.0 was converted!
-- 1.5   | real
-- NULL  | null
Practical tip: Declare columns as NUMERIC or DECIMAL when you want SQLite to store whole numbers as integers and fractional numbers as reals automatically — without you having to decide ahead of time.
Key takeaway

What to remember about SQLite data types

  • → 5 storage classes on disk: NULL, INTEGER, REAL, TEXT, BLOB
  • → Declared type → affinity (a hint, not a hard rule in normal mode)
  • → BOOLEAN = INTEGER (1 or 0)
  • → Store dates as ISO TEXT '2024-06-01'
  • → Never store money as REAL — use INTEGER (pence/cents)
  • → Add STRICT to opt into hard type enforcement (SQLite 3.37+)
  • → Use typeof(val) to see what was actually stored

Constraint 1 of 6 NOT NULL

🚫 NOT NULL

The simplest constraint — this column must always have a value. If you insert or update a row without providing this column, SQLite refuses with an error.

Required field No NULL allowed Column-level only Pair with DEFAULT
What it does

Rejects any INSERT or UPDATE that leaves this column empty

✓ Valid — all required fields given
INSERT INTO users
  (name, email)
VALUES
  ('Alice', 'a@x.com');
-- ✓ both provided
✗ Fails — email NOT NULL
INSERT INTO users
  (name)
VALUES
  ('Alice');
-- Error: NOT NULL failed: users.email
Full example

Mark every required column

CREATE TABLE employees (
  id          INTEGER  PRIMARY KEY,
  first_name  TEXT     NOT NULL,    -- required
  last_name   TEXT     NOT NULL,    -- required
  department  TEXT     NOT NULL,    -- required
  salary      INTEGER  NOT NULL,    -- required
  middle_name TEXT,                  -- optional
  notes       TEXT                   -- optional
);

-- ✓ Works
INSERT INTO employees
  (first_name, last_name, department, salary)
VALUES ('Sarah', 'Connor', 'Engineering', 7500000);

-- ✗ Fails — salary is NOT NULL
INSERT INTO employees
  (first_name, last_name, department)
VALUES ('John', 'Doe', 'Sales');
-- Error: NOT NULL constraint failed: employees.salary
Rule of thumb: If a row doesn't make sense without this column, add NOT NULL. If it's optional metadata (bio, phone, nickname), leave it nullable. When unsure, prefer NOT NULL — it's easier to loosen than to clean up NULLs in production.
NOT NULL + DEFAULT: The best combo. NOT NULL DEFAULT 'draft' means "required, but if you don't provide it, use this fallback". You skip the column in INSERT and still never get NULL.
Constraint 2 of 6 UNIQUE

💠 UNIQUE

No two rows can have the same value in this column. Perfect for email addresses, usernames, product codes — anything that should only appear once. Multiple NULLs are still allowed.

No duplicates NULLs exempt Single or multi-column Indexed automatically
Single column

Every email and username must be different

CREATE TABLE users (
  id       INTEGER PRIMARY KEY,
  username TEXT    NOT NULL UNIQUE,  -- unique!
  email    TEXT    NOT NULL UNIQUE,  -- unique!
  phone    TEXT    UNIQUE            -- unique if given, NULLs ok
);

-- ✓ First user
INSERT INTO users (username, email)
VALUES ('alice99', 'alice@mail.com');

-- ✗ Duplicate email!
INSERT INTO users (username, email)
VALUES ('alice_x', 'alice@mail.com');
-- Error: UNIQUE constraint failed: users.email

-- ✓ Two users with no phone — NULL ≠ NULL so both allowed
INSERT INTO users (username, email) VALUES
  ('bob',   'bob@mail.com'),
  ('carol', 'carol@mail.com');
-- ✓ Both phone = NULL — fine!
Multi-column UNIQUE

Make a pair of columns unique together

A student can enrol in the same course only once. Either column alone can repeat — only the combination must be unique:

CREATE TABLE enrolments (
  student_id INTEGER NOT NULL,
  course_id  INTEGER NOT NULL,
  enrolled   TEXT,
  UNIQUE (student_id, course_id)  -- pair is unique
);

INSERT INTO enrolments VALUES
  (1, 101, '2024-09-01'),  -- ✓ Alice in SQL
  (1, 102, '2024-09-01'),  -- ✓ Alice in Python (same student, diff course)
  (2, 101, '2024-09-02');  -- ✓ Bob in SQL (diff student, same course)

-- ✗ Alice already enrolled in SQL!
INSERT INTO enrolments VALUES
  (1, 101, '2024-11-01');
-- Error: UNIQUE constraint failed: enrolments.student_id, enrolments.course_id
Performance bonus: UNIQUE creates an index automatically — so queries filtering by that column are fast, even without a manual CREATE INDEX.
Constraint 3 of 6 PRIMARY KEY

🔑 PRIMARY KEY

Every row needs a way to be uniquely identified — that's the primary key. Think of it as a passport number: every person has exactly one, no two people share the same, and it never changes.

Uniquely identifies a row Cannot be NULL Implicit UNIQUE + index One per table
What is it and why does it matter?

The address of a row — how you find it later

Without a primary key, how would you update or delete one specific row? If two users are both named "Alice", WHERE name = 'Alice' hits both. A primary key gives every row an unambiguous address — you always know exactly which row you're talking about.

Without PRIMARY KEY

  • Two "Alice" rows — which one do you update?
  • DELETE hits all matching rows
  • No fast lookup by row identity
  • Can't reliably reference from other tables

With PRIMARY KEY

  • Every row has a unique, stable id
  • UPDATE / DELETE by id is always exact
  • Automatic index — fast lookups
  • Other tables can safely reference it
Rules: A PRIMARY KEY is always UNIQUE and always NOT NULL — SQLite enforces both automatically. You only need one per table.
SQLite special behaviour

INTEGER PRIMARY KEY — the rowid alias trick

SQLite internally numbers every row with a hidden 64-bit integer called the rowid. When you declare a column as INTEGER PRIMARY KEY, that column becomes an alias for the rowid. This means:

  • → If you omit the id column in INSERT, SQLite picks the next available integer automatically
  • → This is not the same as AUTOINCREMENT — it reuses deleted IDs (explained below)
  • → If you supply the id, SQLite uses that value (and fails if it already exists)
CREATE TABLE orders (
  id          INTEGER PRIMARY KEY,  -- alias for rowid; auto-assigned when omitted
  customer    TEXT    NOT NULL,
  total_pence INTEGER NOT NULL
);

-- Omit id — SQLite assigns the next available rowid
INSERT INTO orders (customer, total_pence) VALUES
  ('Alice', 1499),   -- gets id = 1
  ('Bob',   2250),   -- gets id = 2
  ('Carol', 899);    -- gets id = 3

-- Check the last assigned id
SELECT last_insert_rowid();  -- → 3

-- Supply id manually — fine if it doesn't exist yet
INSERT INTO orders (id, customer, total_pence)
VALUES (99, 'Zara', 5000);  -- ✓ id = 99

-- ✗ Duplicate PK — always fails
INSERT INTO orders (id, customer, total_pence)
VALUES (1, 'Hacker', 0);
-- Error: UNIQUE constraint failed: orders.id
When you need guaranteed monotonic IDs

AUTOINCREMENT — the explicit keyword

Adding the AUTOINCREMENT keyword changes one thing: deleted IDs are never reused. SQLite tracks the highest ID ever used in a special internal table (sqlite_sequence) and always goes higher.

-- With AUTOINCREMENT: IDs only ever go up, never reused
CREATE TABLE audit_log (
  id         INTEGER PRIMARY KEY AUTOINCREMENT, -- strict monotonic
  action     TEXT    NOT NULL,
  user_id    INTEGER NOT NULL,
  happened   TEXT    NOT NULL DEFAULT (datetime('now'))
);

INSERT INTO audit_log (action, user_id) VALUES
  ('login',  5),  -- id = 1
  ('update', 5),  -- id = 2
  ('logout', 5);  -- id = 3

-- Delete id = 2
DELETE FROM audit_log WHERE id = 2;

-- Next insert gets id = 4, NOT 2 (2 is gone forever)
INSERT INTO audit_log (action, user_id)
VALUES ('login', 7);  -- id = 4

INTEGER PRIMARY KEY

  • ✓ Auto-assigns next available rowid
  • ~ Deleted IDs can be reused
  • ✓ Faster — no extra table lookup
  • → Good for 99% of apps

INTEGER PRIMARY KEY AUTOINCREMENT

  • ✓ Auto-assigns, strictly monotonic
  • ✓ Deleted IDs are never reused
  • ~ Slightly slower (sqlite_sequence lookup)
  • → Use for audit logs, event streams
When one column isn't enough

Composite PRIMARY KEY — two columns together

When no single column uniquely identifies a row, declare the key across multiple columns. Common in junction tables (many-to-many relationships):

-- A user can have a tag only once
-- user_id=1, tag='admin' is a unique combination
CREATE TABLE user_tags (
  user_id  INTEGER NOT NULL,
  tag      TEXT    NOT NULL,
  added_at TEXT    DEFAULT (date('now')),
  PRIMARY KEY (user_id, tag)  -- the PAIR is the key
);

INSERT INTO user_tags (user_id, tag) VALUES
  (1, 'admin'),       -- ✓
  (1, 'developer'),   -- ✓ same user, different tag
  (2, 'admin');        -- ✓ different user, same tag

INSERT INTO user_tags (user_id, tag)
VALUES (1, 'admin');    -- ✗ (1, 'admin') already exists!
-- Error: UNIQUE constraint failed: user_tags.user_id, user_tags.tag
Note: Composite PKs cannot use the rowid alias trick, so IDs are not auto-assigned. You must always supply both columns explicitly.
Constraint 4 of 6 FOREIGN KEY

🔗 FOREIGN KEY

The constraint that connects tables together. A foreign key says "this value must exist in another table". It prevents orphan records — orders without customers, comments without posts.

Referential integrity OFF by default! PRAGMA foreign_keys = ON ON DELETE CASCADE

Basic usage

Always enable PRAGMA foreign_keys first!

-- ⚠️  Run this at the START of every connection
PRAGMA foreign_keys = ON;

CREATE TABLE customers (
  id    INTEGER PRIMARY KEY,
  name  TEXT    NOT NULL,
  email TEXT    NOT NULL UNIQUE
);

CREATE TABLE orders (
  id          INTEGER PRIMARY KEY,
  customer_id INTEGER NOT NULL,
  product     TEXT    NOT NULL,
  total_pence INTEGER NOT NULL,
  FOREIGN KEY (customer_id) REFERENCES customers(id)
);

INSERT INTO customers (name, email)
VALUES ('Alice', 'alice@shop.com');  -- id = 1

-- ✓ customer 1 exists
INSERT INTO orders (customer_id, product, total_pence)
VALUES (1, 'Laptop', 89999);

-- ✗ customer 999 doesn't exist!
INSERT INTO orders (customer_id, product, total_pence)
VALUES (999, 'Keyboard', 4999);
-- Error: FOREIGN KEY constraint failed
🔴 SQLite quirk — FK is OFF by default! For backwards-compatibility, SQLite ships with foreign key enforcement disabled. You MUST run PRAGMA foreign_keys = ON; at every connection start. Put it in your app's database initialization.
Cascade actions

Control what happens when the parent row changes

-- ON DELETE CASCADE: delete parent → delete all children
CREATE TABLE orders (
  id          INTEGER PRIMARY KEY,
  customer_id INTEGER NOT NULL,
  product     TEXT,
  FOREIGN KEY (customer_id) REFERENCES customers(id)
    ON DELETE CASCADE    -- auto-delete orders when customer is deleted
    ON UPDATE CASCADE    -- auto-update FK if customer id changes
);

-- ON DELETE RESTRICT: prevent deleting if child rows exist
FOREIGN KEY (customer_id) REFERENCES customers(id)
  ON DELETE RESTRICT    -- error if you try to delete a customer who has orders

-- ON DELETE SET NULL: orphan rows point to NULL instead
FOREIGN KEY (customer_id) REFERENCES customers(id)
  ON DELETE SET NULL    -- order stays, customer_id becomes NULL

-- Deleting a customer cascades to all their orders
DELETE FROM customers WHERE id = 1;
-- All orders with customer_id = 1 are also deleted!
Constraint 5 of 6 CHECK

✅ CHECK

Write any boolean expression — the database enforces it. Prices must be positive. Ratings must be 1–5. Status must be one word from a list. This is where constraints get creative.

Any boolean expression Column or table level Ranges, enums, cross-column Row rejected if FALSE
Column-level CHECK

Ranges, allowed values, and custom logic

CREATE TABLE products (
  id          INTEGER PRIMARY KEY,
  name        TEXT    NOT NULL,
  price_pence INTEGER NOT NULL
                       CHECK (price_pence >= 0),         -- no negatives
  discount    INTEGER DEFAULT 0
                       CHECK (discount BETWEEN 0 AND 100), -- 0–100 %
  status      TEXT    NOT NULL
                       CHECK (status IN
                         ('draft', 'active', 'archived')),  -- enum!
  rating      REAL
                       CHECK (rating IS NULL              -- nullable
                          OR (rating >= 1 AND rating <= 5)) -- 1–5 range
);

-- ✓ Valid
INSERT INTO products (name, price_pence, status, rating)
VALUES ('Cool Hat', 2499, 'active', 4.5);

-- ✗ Negative price
INSERT INTO products (name, price_pence, status)
VALUES ('Free Stuff', -1, 'active');
-- Error: CHECK constraint failed: price_pence >= 0

-- ✗ Invalid status
INSERT INTO products (name, price_pence, status)
VALUES ('Widget', 999, 'for_sale');
-- Error: CHECK constraint failed

Table-level CHECK (multi-column)

Constraints that compare two columns

CREATE TABLE events (
  id           INTEGER PRIMARY KEY,
  title        TEXT    NOT NULL,
  start_date   TEXT    NOT NULL,
  end_date     TEXT    NOT NULL,
  capacity     INTEGER NOT NULL CHECK (capacity > 0),
  tickets_sold INTEGER DEFAULT 0,
  -- table-level: end must be after start
  CHECK (end_date > start_date),
  -- table-level: can't oversell
  CHECK (tickets_sold <= capacity)
);

-- ✗ end before start
INSERT INTO events (title,start_date,end_date,capacity)
VALUES ('Conf', '2024-12-10', '2024-12-09', 200);
-- Error: CHECK constraint failed: end_date > start_date
Constraint 6 of 6 DEFAULT

🎯 DEFAULT

A fallback value when INSERT doesn't provide one. Literals like 0 or 'draft', or computed expressions like datetime('now') — evaluated at insert time.

Literal or expression Evaluated at insert time Wrap expressions in () Combine with NOT NULL
Why DEFAULT matters

Skip optional columns and still never get NULL

Without DEFAULT — verbose
INSERT INTO posts
  (title, body,
   status, views,
   pinned, created_at)
VALUES
  ('Hello', 'World',
   'draft', 0,
   0, '2024-06-01 10:00');
With DEFAULT — clean
INSERT INTO posts
  (title, body)
VALUES
  ('Hello', 'World');

-- status, views, pinned,
-- created_at all filled
-- in automatically ✓
Full example

Literals, booleans, and timestamp expressions

CREATE TABLE posts (
  id         INTEGER PRIMARY KEY,
  title      TEXT    NOT NULL,
  body       TEXT    NOT NULL,
  status     TEXT    NOT NULL DEFAULT 'draft',           -- literal string
  views      INTEGER NOT NULL DEFAULT 0,                 -- literal 0
  is_pinned  BOOLEAN NOT NULL DEFAULT FALSE,             -- FALSE = 0
  created_at TEXT    NOT NULL DEFAULT (datetime('now')),  -- expression!
  updated_at TEXT    NOT NULL DEFAULT (datetime('now'))   -- expression!
);

-- Minimal INSERT — only required fields
INSERT INTO posts (title, body)
VALUES ('My First Post', 'Hello world!');

SELECT * FROM posts;
-- id:1  title:'My First Post'  body:'Hello world!'
-- status:'draft'  views:0  is_pinned:0
-- created_at:'2024-06-01 09:30:00'

-- Publish — update status and timestamp
UPDATE posts
SET   status     = 'published',
      updated_at = datetime('now')
WHERE id = 1;
Syntax reminder: Wrap expressions in parentheses: DEFAULT (datetime('now')). Plain literals don't need them: DEFAULT 0, DEFAULT 'draft', DEFAULT TRUE.
All 6 constraints — putting it together

A production-ready schema

📦 e-commerce schema — all 6 constraints at once
PRAGMA foreign_keys = ON;  -- always!

CREATE TABLE customers (
  id        INTEGER PRIMARY KEY,              -- 🔑 PK
  name      TEXT    NOT NULL,                 -- 🚫 required
  email     TEXT    NOT NULL UNIQUE,          -- 🚫💠 required + unique
  tier      TEXT    NOT NULL DEFAULT 'free'   -- 🎯 default
                     CHECK (tier IN           -- ✅ enum
                       ('free','pro','enterprise')),
  credit    INTEGER NOT NULL DEFAULT 0        -- 🎯 default
                     CHECK (credit >= 0),      -- ✅ no negative
  joined_at TEXT    NOT NULL DEFAULT (date('now')) -- 🎯 auto date
);

CREATE TABLE orders (
  id          INTEGER PRIMARY KEY,
  customer_id INTEGER NOT NULL,
  status      TEXT    NOT NULL DEFAULT 'pending' -- 🎯
                       CHECK (status IN         -- ✅
                         ('pending','paid','shipped','delivered','cancelled')),
  total_pence INTEGER NOT NULL CHECK (total_pence > 0), -- ✅
  placed_at   TEXT    NOT NULL DEFAULT (datetime('now')),-- 🎯
  FOREIGN KEY (customer_id) REFERENCES customers(id) -- 🔗
    ON DELETE RESTRICT
);
🎉 With these constraints it is impossible to insert a negative credit balance, an order for a missing customer, a duplicate email, an invalid tier, or a negative total — all without a single line of application validation code.

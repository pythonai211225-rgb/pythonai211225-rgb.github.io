Auto Increment

Auto increment helps us create IDs automatically. It saves time and prevents manual ID mistakes

Why do we need it

  • You do not need to choose a new ID every time
  • Each new row gets a unique number automatically
  • It avoids duplicate-ID bugs in application code

How it works in SQLite

In SQLite, the common pattern is INTEGER PRIMARY KEY. If you do not provide the ID, SQLite gives the next available rowid

Common pattern
CREATE TABLE students (
  id INTEGER PRIMARY KEY,
  name TEXT NOT NULL
);

INSERT INTO students (name) VALUES ('Dana');
INSERT INTO students (name) VALUES ('Omar');

SELECT id, name FROM students;
-- result:
-- 1 | Dana
-- 2 | Omar

INTEGER PRIMARY KEY vs AUTOINCREMENT

Both create automatic IDs, but behavior is a little different

Option Behavior When to use
INTEGER PRIMARY KEY Fast and simple, may reuse old deleted IDs in some cases Default choice for most apps
INTEGER PRIMARY KEY AUTOINCREMENT Never reuses older IDs, always moves forward When ID reuse must never happen
AUTOINCREMENT is stricter but can be a little slower, so use it only when you need no reuse

AUTOINCREMENT example

Example
CREATE TABLE orders (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  item_name TEXT NOT NULL
);

INSERT INTO orders (item_name) VALUES ('Keyboard');
INSERT INTO orders (item_name) VALUES ('Mouse');

SELECT id, item_name FROM orders;
-- result:
-- 1 | Keyboard
-- 2 | Mouse

DELETE FROM orders WHERE id = 2;
-- delete last row

INSERT INTO orders (item_name) VALUES ('Headset');
-- with AUTOINCREMENT, SQLite does not reuse id 2

SELECT id, item_name FROM orders;
-- result:
-- 1 | Keyboard
-- 3 | Headset

Quick memory tips

  • Use INTEGER PRIMARY KEY for most tables
  • Use AUTOINCREMENT only if IDs must never be reused
  • Let SQLite generate IDs instead of manual typing

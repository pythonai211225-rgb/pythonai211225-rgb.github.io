The Design Process

Follow these 5 steps every time — even for small databases. Skipping ahead to SQL always creates problems.

1

Identify Entities

Look for the nouns in the business description — people, places, things, events. Each becomes a table.

Customer · Order · Product · Category · Invoice
2

Define Attributes

For each entity, list its properties. Each property becomes a column. Ask: does this belong to this entity, or to another one?

Customer: name, email, phone  |  Product: name, price, stock
3

Map Relationships

Look for the verbs: "a customer places orders", "an order contains products". Decide the cardinality: 1:1, 1:N, or N:M.

places → 1:N  |  contains → N:M (needs junction table)
4

Apply Normal Forms

Check each table against 1NF, 2NF, and 3NF. Split tables that store more than one concept, move repeated data into lookup tables.

No repeating groups · No partial deps · No transitive deps
5

Draw Before You Code

Sketch the Entity-Relationship Diagram (ERD). Draw tables, columns, and relationship lines. Only then open your SQL editor.

Boxes = tables · Lines = relationships · Crow's foot = many

Normal Forms — Quick Reminder

Apply these rules in order while designing. Each form builds on the previous one.

1NF
Atomic values.
Every cell holds one value. No lists, no comma-separated tags, no repeating columns (phone1, phone2…).
→
2NF
Full key dependency.
Every non-key column depends on the entire primary key — not just part of it (matters most with composite PKs).
→
3NF
No transitive deps.
Non-key columns must depend only on the key, not on each other. If column B determines column C, move them to their own table.

Design Checklist

Click each item to check it off as you review your schema before writing SQL.

Every table has a clear Primary Key that uniquely identifies each row
No cell contains multiple values — all values are atomic (1NF)
Every column describes only the entity of its table — no unrelated data mixed in
No partial dependencies — non-key columns depend on the full PK (2NF)
No transitive dependencies — non-key columns don't determine other non-key columns (3NF)
Every Foreign Key is on the "many" side of a 1:N relationship
Every Many-to-Many relationship is resolved with a junction table holding two FKs
You have drawn an ERD diagram before writing any CREATE TABLE statements

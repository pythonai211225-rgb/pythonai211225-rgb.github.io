The Research Approach

Every SQL research task follows the same 3-step mental process. Master this and you can answer almost any business question with SQL.

1

Understand the Question

Read it slowly. Who are we looking for? What is the condition? What data do we need? Rephrase it in plain English before touching SQL.

"Find customers refunded more than once" → count refunds per customer, keep only > 1
2

Map to Tables & Columns

Identify which tables hold the data you need. Draw the path between them — which foreign keys connect them?

refunds → order_id → orders → customer_id → customers
3

Build the SQL Clause by Clause

Start with SELECT and FROM. Add JOINs one at a time. Add WHERE if needed. Add GROUP BY. Add HAVING for the count condition. Add ORDER BY last.

SELECT → FROM → JOIN → GROUP BY → HAVING → ORDER BY

Examples

🏆
Example 1 — Top Customers
Find the top 5 customers by total spending, with their order count. Uses JOIN, SUM, COUNT, GROUP BY, ORDER BY, and LIMIT together.
SUM · COUNT · ORDER BY · LIMIT
🔄
Example 2 — Refund Analysis
Find all orders that were refunded more than once — same deal, paid back twice. Uses GROUP BY + HAVING COUNT on the refunds table, with JOINs for customer names.
GROUP BY · HAVING · JOIN
🚨
Example 3 — Suspicious Logins
Detect hacked employee accounts using impossible travel: two logins from different countries within seconds. Uses self-join and time arithmetic.
Self-Join · strftime · Security

Skills You Need for Research Queries

Make sure you're comfortable with these before diving in.

JOIN — connecting two tables on a shared column (foreign key)
GROUP BY — grouping rows that share the same value in a column
COUNT(*) — counting rows in each group
HAVING — filtering groups after GROUP BY (like WHERE but for groups)
SUM / AVG — aggregating numeric values per group
ORDER BY … DESC LIMIT N — ranking and taking the top N results

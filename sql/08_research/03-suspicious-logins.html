Example 3 — Security

Suspicious Login Detection

The security team got an alert. An employee logged in from Israel in the morning — then the same evening a second login appeared from Russia. Same account. Same day. Different continent.

The Scenario

🚨
Security Alert
An employee account logged in from two different countries on the same day. No one flies Tel Aviv → Moscow and back in one workday — one of those sessions is an attacker.

This attack pattern is called Impossible Travel. A real person cannot be in Tel Aviv in the morning and Moscow in the evening of the same day — not without a supersonic jet. When the database shows two logins from the same account on the same calendar day but from countries thousands of kilometres apart, one of those logins is stolen. Your job: write SQL to catch it.

The Question

Security Question
🔐 Which employees had two logins from different countries on the same day? Show the employee name, their home country, both login locations, and the shared login date.

Real security platforms (like Azure AD, Okta, and AWS IAM) run exactly this kind of query. A same-day cross-country check is a practical starting point — a stolen credential used from another country on the same calendar day is a strong signal of compromise, especially when the countries are on different continents.

The Schema

Two tables: employees stores who works here and where they live; login_logs records every login event with timestamp and country.

Think It Through — Step by Step

1
What are we comparing?
We need two login rows for the same employee — but different countries and close in time. A single table row can't tell us that. We need to compare the table against itself: a self-join.
2
How does a self-join work?
We alias login_logs twice: l1 and l2. Join them on the same employee_id, then filter so l2.login_time is after l1.login_time and the countries are different.
3
How do we check that both logins are on the same calendar day?
strftime('%Y-%m-%d', login_time) extracts just the date part (year-month-day) from a datetime string. If both logins return the same date string, they happened on the same day.
4
How do we show the employee's home country?
JOIN employees to bring in e.country. That lets us also confirm whether the suspicious login came from outside the employee's normal home country.

Step 1 — Self-Join: Find Same-Day Cross-Country Logins

Start without the employees table. Pair up login rows for the same account where the country changed but the calendar date is the same.

SQL
SELECT   l1.employee_id,
         l1.country    AS first_country,
         l1.login_time AS first_time,
         l2.country    AS second_country,
         l2.login_time AS second_time,
         strftime('%Y-%m-%d', l1.login_time) AS login_date
FROM     login_logs l1
JOIN     login_logs l2
         ON  l2.employee_id = l1.employee_id
         AND l2.login_time  > l1.login_time
         AND l2.country    != l1.country
WHERE    strftime('%Y-%m-%d', l1.login_time)
         = strftime('%Y-%m-%d', l2.login_time)
ORDER BY l1.login_time;

Step 2 — Add JOIN for Employee Details

Now attach the employees table to show the person's name and their registered home country alongside the suspicious pair of logins.

SQL
SELECT   e.name,
         e.country                                        AS home_country,
         l1.country                                       AS login_from,
         l1.login_time                                    AS first_login,
         l2.country                                       AS suspicious_from,
         l2.login_time                                    AS second_login,
         strftime('%Y-%m-%d', l1.login_time)              AS login_date
FROM     login_logs l1
JOIN     login_logs l2
         ON  l2.employee_id = l1.employee_id
         AND l2.login_time  > l1.login_time
         AND l2.country    != l1.country
JOIN     employees e ON e.id = l1.employee_id
WHERE    strftime('%Y-%m-%d', l1.login_time)
         = strftime('%Y-%m-%d', l2.login_time)
ORDER BY l1.login_time;

How the Self-Join Works

A self-join treats one table as if it were two separate tables by using two different aliases. Here's how each join condition eliminates unwanted pairs:

1

l2.employee_id = l1.employee_id

Only pair rows that belong to the same employee. Without this, you'd be comparing Yael's logins against James's logins — meaningless.

2

l2.login_time > l1.login_time

Ensure l1 is always the earlier login and l2 is later. Without this you'd get each pair twice (A→B and B→A) and also compare a row with itself.

3

l2.country != l1.country

Only keep pairs where the country changed. Logging in twice from the same country is normal. We only care about cross-country jumps.

4

WHERE strftime('%Y-%m-%d', l1…) = strftime('%Y-%m-%d', l2…)

The date check is in WHERE, not in the JOIN — because it's a derived value. strftime('%Y-%m-%d', ...) strips the time part, leaving only the date. Equality means both logins happened on the same calendar day.

Key Takeaways

💡

Self-Join: compare a table to itself

Alias the same table twice (login_logs l1, login_logs l2). This lets you compare two rows from the same table — essential when the relationship lives within the same table (before/after, parent/child, suspicious pairs).

💡

strftime('%Y-%m-%d', …) for date extraction

SQLite stores datetimes as text. strftime('%Y-%m-%d', datetime_string) strips away the time part and returns just the date — '2024-03-15'. Comparing two of these checks whether both logins happened on the same calendar day.

💡

This is real-world security SQL

Impossible travel detection is used by every major identity provider. The SQL you wrote here is the same logic behind the "suspicious login" email you've probably received from Google or GitHub. Pattern: self-join + time gap + geo mismatch.

What is NoSQL?

NoSQL databases store data differently — not in rows and columns, but as documents, key-value pairs, or graphs. They solve specific problems that relational databases weren't designed for

NoSQL = "Not Only SQL" 4 main types Not a replacement — a different tool
The right framing

SQL and NoSQL are tools, not rivals

A common mistake is thinking NoSQL is "better" or "newer" than SQL. It's neither. They solve different problems. Most real companies use both — a relational database for structured data and a NoSQL store for something else alongside it

Instagram uses PostgreSQL (SQL) for user accounts and posts, and Redis (NoSQL key-value) to cache timelines so they load in milliseconds. MongoDB (NoSQL document) is popular for content management where every article has a different shape

✓ Pick SQL when…

  • Your data has a clear, fixed structure (users, orders, products)
  • You need to query across multiple tables with joins
  • Data accuracy and consistency are critical (finance, healthcare)
  • Your team knows the schema upfront
  • You need complex filtering, aggregation, or reporting

✓ Pick NoSQL when…

  • ✓Each record can have different fields (flexible schema)
  • ✓You need to cache data at extreme speed (sub-millisecond reads)
  • ✓Relationships between items are more important than the items themselves (social graphs)
  • ✓Storing massive, horizontal scale (billions of rows across many servers)
  • ✓Schema changes constantly during early product development
The four families

Four types of NoSQL databases

📄

Document Store

Examples: MongoDB, CouchDB, Firestore

Stores data as JSON-like documents. Each document is self-contained and can have its own fields — no fixed schema. Perfect when every record is slightly different (think: a product catalogue where a T-shirt has "size" but a laptop has "RAM")

// A product document in MongoDB
{
  "_id": "prod_001",
  "name": "Wireless Headphones",
  "price": 89.99,
  "specs": {
    "battery_hours": 30,
    "bluetooth": "5.2",
    "colours": ["black", "white", "navy"]
  }
}

Notice: no fixed columns. A book document in the same collection could have "author" and "ISBN" — fields the headphones don't need

🗝️

Key-Value Store

Examples: Redis, DynamoDB, Memcached

The simplest model: a giant dictionary. Give it a key, get back a value. Blindingly fast because there's nothing to compute — it just looks up the key. Used for caching, sessions, real-time leaderboards

# Redis commands
SET  session:user_42   "{ name: Alice, cart: [101,205] }"   EX 3600
GET  session:user_42
# → "{ name: Alice, cart: [101,205] }"  — returned in <1ms

SET  rate_limit:ip_1_2_3_4   47   EX 60
# Tracks how many API calls this IP made in the last 60 seconds
🕸️

Graph Database

Examples: Neo4j, Amazon Neptune

Stores nodes (things) and edges (relationships between things) as first-class citizens. When the relationships are as important as the data, graph databases win. Used for social networks, fraud detection, recommendation engines

Example query: "Find all friends of Alice's friends who live in London and like jazz" — in SQL this needs 4–5 expensive joins. In a graph database it's a natural traversal along edges, and it stays fast even at Facebook scale
📊

Wide-Column Store

Examples: Apache Cassandra, Google Bigtable, HBase

Like a table but with potentially millions of columns, and each row can have different columns. Optimised for writing huge volumes of data fast and reading it back by a known key. Used for time-series data (sensor readings, log streams, IoT)

Real use: Apple stores over 10 petabytes of iCloud data in Cassandra. Netflix uses it to store viewing history for 270 million subscribers. SQL couldn't horizontally scale across thousands of servers the same way
Quick guide

Which to reach for?

Situation Reach for Why
User accounts, orders, invoices'SQL'Structured, relational, accuracy critical
Product catalogue with varied attributes'Document (MongoDB)'Flexible schema per item
User sessions & login tokens'Key-Value (Redis)'Sub-ms lookup, auto-expiry
"Who follows whom" on a social app'Graph (Neo4j)'Relationships are the data
IoT sensor readings every second'Wide-column (Cassandra)'Massive write throughput
A small app you're learning on'SQLite'Simple, zero setup, teaches SQL
